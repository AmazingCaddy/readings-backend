# 系统设计常见术语表

这篇是系统设计案例的词典。读案例时如果遇到不熟悉的词，可以先回到这里看：它是什么、为什么会出现、大概怎么实现，以及在哪些案例里会用到。

## 读扩散 / 写扩散

这两个词常见于 Feed、通知、粉丝时间线这类系统。

**写扩散** 是在写入时就把数据推给所有接收者。比如 A 发微博，系统立刻把这条微博 ID 写进所有粉丝的首页时间线。

```text
A 发微博 -> 查 A 的粉丝 -> 写入粉丝 B/C/D 的 feed 列表
```

优点是粉丝刷首页很快；缺点是大 V 发一条内容会产生巨大写入量。

**读扩散** 是写入时只保存作者自己的内容，读者刷新时再临时聚合关注对象的内容。

```text
B 刷首页 -> 查 B 关注的人 -> 分别读取最近内容 -> 合并排序
```

优点是发文轻；缺点是读首页变重。微博 Feed 常用混合策略：普通用户写扩散，大 V 读扩散。

## Feed / 时间线

Feed 是按某种规则排列的一组内容流，比如微博首页、朋友圈、推荐流。时间线是 Feed 的一种常见形式，通常按时间或排序分数从新到旧排列。

实现上，时间线一般不直接存完整正文，而是存内容 ID 列表。读取时再批量查询正文、作者信息、互动计数和权限状态。

## Fanout

Fanout 的意思是“一次输入扩散成多次输出”。微博里，作者发一条微博后，把这条微博分发给很多粉丝，就是 fanout。

Fanout 常用 MQ 和 worker 异步执行，因为它可能很慢，不应该阻塞用户发布请求。

## 削峰

削峰是把瞬时高流量变得平滑。比如秒杀开始时有 100 万请求同时进来，但订单系统每秒只能处理 1 万个，就用限流、队列、排队页把流量摊开。

常见实现：网关限流、MQ 队列、令牌桶、排队系统、异步 worker。

## 回源

回源是缓存没命中或缓存不可用时，请求回到权威数据源读取。比如 Redis 没有商品详情，就回源数据库查询。

回源本身正常，但高并发下大量请求同时回源会打垮数据库，所以需要缓存击穿、雪崩和穿透保护。

## 热点 / 热 Key

热点是访问极度集中的数据或对象。热 Key 是 Redis 中被大量请求访问的单个 key，比如大促活动配置、热门商品详情、热点微博计数。

热 Key 即使命中缓存，也可能把 Redis 单分片或应用连接池打满。常见处理：本地缓存、key 拆分、读副本、限流、返回旧值。

## 最终一致性

最终一致性是指系统不保证所有地方立刻一致，但经过一段时间和重试补偿后会一致。

微博 Feed 可以接受最终一致：发文后粉丝首页晚几秒看到通常可以接受。支付成功和出票则不能随便接受长时间不一致，需要更强的状态校验和补偿。

## 幂等

幂等是同一个操作执行一次和执行多次，最终效果一致。比如支付回调重复来了 3 次，订单也只能从待支付变成已支付一次。

常见实现：幂等键、唯一索引、消息去重表、状态条件更新。

## 补偿

补偿是在某个步骤失败或不确定后，通过后续任务把系统修回正确状态。

例子：支付回调丢了，查单任务从支付渠道查到已支付，然后补写本地支付状态；占座成功但订单创建失败，补偿任务释放座位。

## 状态机

状态机是把业务对象的状态和合法流转规则写清楚。订单不能从已取消直接变成已支付，支付单不能从已退款回到支付中。

状态机通常配合条件更新：

```sql
update orders
set status = 'PAID'
where id = ? and status = 'PENDING_PAYMENT';
```

只有当前状态符合预期时才允许推进。

## Outbox

Outbox 是一种可靠发消息模式。服务在同一个数据库事务里写业务表和 outbox 事件表，然后由后台发布器把事件发到 MQ。

它解决的是“数据库写成功，但消息发送失败”的问题。代价是消息通常至少发送一次，所以消费者必须幂等。

## DLQ

DLQ 是 Dead Letter Queue，死信队列。消息消费多次失败或遇到不可重试错误时，不再无限重试，而是放入 DLQ 等待排查和重放。

DLQ 的关键不是“有一个队列”，而是要有告警、失败上下文、修复流程和安全重放工具。

## SLI / SLO

SLI 是衡量服务体验的指标，比如订单创建成功率、P99 延迟。SLO 是目标，比如“99.9% 订单创建请求在 800ms 内成功”。

SLO 告警通常围绕错误预算燃烧速度，而不是 CPU 或内存这种低层指标直接 page。

## P99

P99 是第 99 百分位延迟。如果某接口 P99 是 900ms，表示 99% 请求在 900ms 内完成，最慢的 1% 超过 900ms。

P99 比平均延迟更能暴露长尾问题，比如连接池等待、锁等待、慢 SQL、GC 暂停和下游抖动。

## 预扣库存

预扣库存是先在高性能存储中扣掉一个库存名额，再异步创建正式订单。秒杀里常用 Redis 预扣库存，避免所有请求直接打数据库。

预扣不是最终正确来源，后面还要有订单落库、支付超时释放和库存对账。

## 逻辑过期

逻辑过期是缓存值里带一个过期时间，而不是完全依赖 Redis TTL。读请求发现逻辑过期时，可以先返回旧值，再异步刷新。

它常用于热点数据，避免热点 key 到期瞬间大量请求同时回源数据库。

## 游标分页

游标分页是用上一页最后一条记录的位置继续查询，而不是用 `offset` 跳过很多行。

```sql
where created_at < ? or (created_at = ? and id < ?)
order by created_at desc, id desc
limit 20
```

它适合大表、Feed、订单列表和无限滚动，因为深分页性能更稳定，也不容易重复或漏数据。

## 读写分离

读写分离是写请求走主库，读请求走只读副本或读模型。它能提升读吞吐，但会带来复制延迟。

适合查询量远大于写入量的系统，例如余票查询、订单列表、Feed 读取。对强一致读，要回到主库或设计读己之写策略。

## 读模型 / 写模型

写模型负责保证业务状态正确，读模型负责让查询更快。比如订单写入时以订单表和状态机为准，订单详情页可以用一个聚合好的读模型快速展示。

读模型通常通过 MQ、CDC 或后台任务从写模型同步，天然是最终一致。

## 令牌桶 / 漏桶

令牌桶和漏桶都是限流算法。令牌桶允许一定突发流量，只要桶里有令牌就放行；漏桶更像固定速率排水，让请求稳定通过。

秒杀、购票和支付接口通常会使用限流保护下游，避免系统在流量尖峰下整体崩溃。

## 对账

对账是把本地账务记录和外部渠道账单进行比对。支付系统里，即使回调和查单都做了，也仍然需要对账发现资金差异。

对账发现差异后，要生成差错记录，进入自动补偿或人工处理流程。

## 推荐阅读方式

读系统设计案例时，可以按这个顺序理解：

1. 先找到系统最怕什么：读太多、写太多、库存不能错、资金不能错，还是消息不能丢。
2. 再看核心数据模型：哪些表是权威数据，哪些只是缓存或读模型。
3. 再看关键流程图：请求先经过哪些保护，再落到数据库或 MQ。
4. 最后看失败补偿：如果某一步失败，系统靠什么恢复正确。
