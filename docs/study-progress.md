# 我的后端学习档案

这份文档是学习总览页，用来记录学习目标、路线、当前进度和每节课的入口。每节课的详细问答、背后解释、面试表达和复盘记录，拆分到独立的课程笔记里，避免总览页越来越大。

## 当前学习画像

- **背景**：已有前端/客户端经验，熟悉接口调用、页面状态、网络请求和用户侧体验。
- **目标方向**：转向后端开发，重点掌握高并发、高性能、高可靠系统的工程机制。
- **学习偏好**：需要引导式、启发式学习，需要实战场景驱动，不只背概念。
- **最终目的**：能在面试中讲清楚后端项目设计、性能排查、可靠性设计和线上问题处理思路。

## 学习协作方式

学习过程中可以随时提问，不需要等一篇文章读完。每次讨论一个问题后，我会尽量把它沉淀成可复习的内容，避免知识点只停留在当次对话里。

常用节奏是：

1. **先摸底**：确认当前理解、项目经验和易混点。
2. **再搭桥**：把新概念连接到熟悉的前端/客户端经验，比如请求、状态、缓存、网络错误和用户体验。
3. **讲核心**：抓住这个主题最重要的机制、边界和失败模式。
4. **给场景**：用商品详情、下单、支付回调、消息消费等真实后端场景解释。
5. **反问确认**：用小问题检查理解，并根据回答补缺口。
6. **沉淀记录**：把本次问题、短回答、背后解释、实战理解和面试表达更新到对应课程笔记。

## 学习目标

### 第一目标：建立后端请求视角

能从一次请求进入系统开始，讲清楚它经过哪些组件、在哪里排队、在哪里访问依赖、如何设置超时、如何通过日志和 trace 定位问题。

### 第二目标：掌握核心后端机制

能理解并应用数据库索引、事务、锁、连接池、缓存、MQ、幂等、限流、熔断、重试、P99 和可观测性。

### 第三目标：完成一个可讲的实战项目

围绕高并发下单或订单系统，设计 API、表结构、缓存、库存扣减、支付回调、MQ、Outbox、幂等、压测和监控。

### 第四目标：形成面试表达

能用 10 到 15 分钟讲清楚一个项目，并能回答接口变慢、消息重复、缓存失效、数据库锁等待、支付回调重复等常见追问。

## 面试表达模板

### 接口变慢怎么排查

我会先按请求链路拆解，而不是直接猜 SQL 慢。第一步看网关和应用整体指标，比如 QPS、错误率、P95/P99。第二步通过 trace id 看慢请求主要耗时在哪一段，是应用本身、Redis、数据库还是下游 RPC。第三步针对具体依赖看连接池等待、慢查询、缓存命中率、锁等待、超时和重试情况。最后根据根因处理，比如优化索引、增加缓存保护、调整连接池、设置合理超时、限制重试或做降级。
