# 我的后端学习档案

这份文档用来记录我的后端学习目标、阶段进度、实战产出和面试表达。学习路线以“前端/客户端转后端”为起点，先补齐后端核心机制，再通过实战项目串联知识，最后沉淀成面试可讲清楚的项目经验。

## 当前学习画像

- **背景**：已有前端/客户端经验，熟悉接口调用、页面状态、网络请求和用户侧体验。
- **目标方向**：转向后端开发，重点掌握高并发、高性能、高可靠系统的工程机制。
- **学习偏好**：需要实战场景驱动，不只背概念。
- **最终目的**：能在面试中讲清楚后端项目设计、性能排查、可靠性设计和线上问题处理思路。

## 学习协作方式

学习过程中可以随时提问，不需要等一篇文章读完。每次讨论一个问题后，我会尽量把它沉淀成可复习的内容，避免知识点只停留在当次对话里。

### 引导式学习方式

带学时优先采用引导式、启发式节奏。每进入一个新主题，我会先用一两个小问题摸清当前理解，再根据回答决定讲解深度和例子，而不是直接从定义开始灌输。

常用节奏是：

1. **先摸底**：确认你对相关概念、项目经验和易混点的当前理解。
2. **再搭桥**：把新概念连接到你熟悉的前端/客户端经验，比如请求、状态、缓存、网络错误和用户体验。
3. **讲核心**：只抓这个主题最重要的机制、边界和失败模式。
4. **给场景**：用商品详情、下单、支付回调、消息消费等真实后端场景解释。
5. **反问确认**：用小问题检查理解，并根据回答补缺口。
6. **沉淀记录**：把本次问题、核心结论、实战理解和面试表达更新到学习档案。

提问时尽量一次只问一个关键问题。如果某个主题很大，先拆成几个小台阶，避免一下子展开太多。

### 每次学习后的记录项

- **本次问题**：记录这次具体问了什么。
- **你的短回答**：保留你当时的直觉答案，方便回看自己的理解变化。
- **核心结论**：用几句话总结真正需要记住的点。
- **背后解释**：把短回答背后的完整推理补出来，说明为什么这个答案成立、边界在哪里、还要继续追问什么。
- **实战理解**：说明这个知识点在项目中怎么用，容易踩什么坑。
- **面试表达**：整理成面试时可以直接讲出来的回答。
- **待复习问题**：留下 1 到 3 个小问题，用来检查是否真的掌握。

### 记录原则

- 小问题及时记录，大主题阶段性总结。
- 优先记录能迁移到项目和面试里的内容。
- 对已经理解的概念，压缩成关键词和表达模板。
- 对还不熟的概念，保留疑问和下一步学习入口。

## 学习目标

### 第一目标：建立后端请求视角

能从一次请求进入系统开始，讲清楚它经过哪些组件、在哪里排队、在哪里访问依赖、如何设置超时、如何通过日志和 trace 定位问题。

### 第二目标：掌握核心后端机制

能理解并应用数据库索引、事务、锁、连接池、缓存、MQ、幂等、限流、熔断、重试、P99 和可观测性。

### 第三目标：完成一个可讲的实战项目

围绕高并发下单或订单系统，设计 API、表结构、缓存、库存扣减、支付回调、MQ、Outbox、幂等、压测和监控。

### 第四目标：形成面试表达

能用 10 到 15 分钟讲清楚一个项目，并能回答接口变慢、消息重复、缓存失效、数据库锁等待、支付回调重复等常见追问。

### 本节核心理解

前端/客户端通常看到的是“发请求、等响应、渲染页面”。后端需要把这次请求拆成一条链路：网关、应用服务、Redis、MySQL、下游服务、日志、指标和 trace。接口变慢时，不能只猜业务代码或 SQL 慢，而要定位请求到底在哪一段排队、等待、超时或重试。

### 短回答背后的解释

#### 为什么“网络卡顿”是合理第一反应，但不能停在这里

从前端/客户端视角看，页面偶尔慢，最容易先想到网络卡顿，这是合理的，因为用户侧确实会受到弱网、DNS、TCP/TLS 建连、丢包、响应下载和客户端本地处理影响。但后端排查不能只停在“网络慢”，因为一次接口请求是一条链路：客户端、网络、网关、应用服务、Redis、MySQL、下游服务、日志和 trace。任何一段排队、等待或重试，都会让用户感觉慢。所以第一步不是直接下结论，而是先拆链路。

#### 为什么客户端 2s、后端 80ms 时优先怀疑网络或入口链路

客户端看到 2s，说明用户完整等待了 2 秒；后端日志显示 80ms，说明后端应用真正处理请求的时间很短。两者之间差出来的 1920ms，大概率不在业务代码、Redis、MySQL 或下游 RPC 里，而是在请求进入后端前、后端返回后，或者网关/传输层之间。此时应该优先查看客户端 timing、DNS/TCP/TLS、网关日志、负载均衡日志、响应大小和客户端本地处理，而不是一上来优化 SQL。

#### 为什么客户端 2s、后端 1800ms 时要继续拆后端内部

如果客户端总耗时 2s，后端处理已经占了 1800ms，说明主要慢点在后端内部。但“后端接口慢”还不是根因，只是定位范围。下一步要用 trace 或分段日志继续拆：应用计算是否慢，线程池或协程是否排队，Redis 是否 miss 或连接池等待，MySQL 是否慢查询、锁等待或连接池耗尽，下游 RPC 是否超时，是否发生了重试放大。后端排查的关键是继续问“这 1800ms 花在哪里”。

#### 为什么等待数据库连接 1200ms、SQL 80ms 时先看连接池

这类问题很容易被误判成“数据库慢”，但 trace 已经说明真正 SQL 查询只用了 80ms，主要耗时在等待数据库连接。也就是说，请求还没真正开始执行 SQL，就已经在连接池前排队了。常见原因包括连接池太小、并发突然升高、连接泄漏、事务持有连接太久、慢请求占满连接池，或者上游重试把请求量放大。此时优先排查连接池指标、活跃连接数、等待队列、等待超时、事务耗时和连接释放逻辑。

#### 为什么库存服务 RPC 1500ms 时优先看库存链路

trace 的价值是把后端 1800ms 拆成多个阶段。如果应用计算、Redis、MySQL 和响应返回都很短，而库存服务 RPC 占了 1500ms，那么优先怀疑库存链路是合理的。但这还不等于“库存服务代码一定有问题”。RPC 慢可能发生在多个位置：调用方等待 HTTP 连接、网络传输、库存服务网关排队、库存服务内部查库、库存服务调用它自己的下游，或者调用方超时后又重试了一次。下一步要看库存服务自己的 trace、错误率、P95/P99、连接池等待、超时和重试配置。

#### 为什么重试可能让系统更危险

重试不是天然有益，它是在用更多请求换取一次成功机会。对于偶发网络抖动、连接重置、短暂 429/503 这类临时错误，受控重试可能提高成功率。但如果下游已经变慢、连接池已经满、线程已经排队，重试会把一次请求变成两次甚至多次请求，直接增加下游 QPS、连接占用和排队时间。结果可能是原本只有库存服务慢，后来商品服务线程也被占住，网关队列变长，更多请求超时，形成级联故障。所以重试必须有最大次数、短超时、退避、jitter、熔断和幂等保障，并且不是所有错误都应该重试。

#### 为什么平均延迟正常时仍然要看 P99

平均延迟描述的是整体平均水平，但它会把少数极慢请求摊薄。比如 99 个请求都是 50ms，1 个请求是 2s，平均值可能仍然看起来不算糟，但那个 2s 的用户体验是真实存在的。P99 表示最慢 1% 请求的延迟，更适合发现长尾问题。长尾通常来自偶发资源等待，例如连接池排队、锁等待、缓存 miss 后回源、下游 RPC 超时、GC 暂停、重试或热点 key。用户说“偶尔很慢”时，平均延迟往往不够敏感，应该优先看 P95/P99、慢请求 trace 和各依赖分段耗时。

#### 为什么 P99 升高但平均延迟不变时，要看少部分慢请求

如果所有请求都变慢，平均延迟通常也会明显上升。现在 P99 突然升高，但平均延迟变化不大，说明大多数请求仍然正常，只有尾部少部分请求被拖慢。这类问题常见于偶发资源竞争，而不是稳定的业务逻辑变慢。例如某些请求命中缓存 miss、某些商品是热 key、某些 SQL 等锁、某些请求等待连接池、某个下游偶发超时，或者某批请求触发重试。排查时应该抽样最慢的 trace，而不是只看平均指标。

#### 为什么看 trace 要先看每一步的时间

trace 的核心价值不是告诉我们“这个接口慢”，而是把一次请求拆成多个 span，并标出每个阶段的耗时。第一眼看每一步的时间，是为了找到主要耗时段：慢在应用计算、Redis、MySQL、下游 RPC、连接池等待、锁等待，还是重试。找到最长阶段后，还要继续下钻。例如 MySQL 慢要区分 SQL 执行慢、锁等待慢、连接池等待慢；RPC 慢要区分调用方等待连接、网络慢、下游入口排队、下游内部依赖慢。不要只看总耗时，要看耗时分布。

### 本节面试表达补充

#### 一个接口偶尔很慢，怎么排查

我的第一反应不是直接猜 SQL 慢，而是先确认这是长尾问题还是整体变慢。会先看 P95/P99，取出 P99 慢请求日志，找到对应的 trace id。然后对比客户端总耗时、网关日志和后端 access log，先判断慢是在网络/入口链路/客户端侧，还是后端接口内部。如果客户端很慢但后端处理很快，就优先查网络、DNS/TCP/TLS、网关传输和客户端侧；如果后端处理本身也慢，就看 trace 里面每个 span 的时间分布，优先找耗时最长的阶段，再继续拆应用计算、Redis、MySQL、连接池、锁等待、下游 RPC、超时和重试，一层层定位根因。

当前自己的简短回答：先找 P99，取出 P99 的 log，找到慢的 trace id，先排查是否是网络问题还是接口问题；如果是接口问题，排查详细 trace 里面的时间分布，优先看耗时最长的，一步步拆解下去。

如果客户端看到请求耗时 2s，但服务端日志显示接口只处理了 80ms，我会优先怀疑问题不在业务逻辑本身，而在客户端到服务端之间的链路，例如网络波动、DNS/TCP/TLS、网关入口、响应传输或客户端本地处理。接下来我会对比客户端 timing、网关日志、后端 access log 和 trace，确认请求到底是进入后端前慢，还是后端返回后慢。

如果客户端耗时 2s，服务端日志也显示处理了 1800ms，那就说明主要耗时在后端内部。下一步不能只说“接口慢”，而要继续拆分：应用本身是否 CPU 高或线程池排队，Redis 是否 miss 或连接池等待，MySQL 是否慢查询、锁等待或连接池耗尽，下游 RPC 是否超时或重试。排查时最好用 trace 把每个依赖调用的耗时展开。

如果 trace 里显示等待数据库连接 1200ms，但真正 SQL 查询只有 80ms，优先排查数据库连接池，而不是先优化 SQL。因为慢点发生在“拿不到连接”这一层，常见原因包括连接池太小、并发请求过高、连接没有及时归还、事务持有连接太久，或者上游重试导致请求量被放大。

如果 trace 展开后发现库存服务 RPC 用了 1500ms，而应用计算、Redis 和 MySQL 都很短，我会优先排查库存服务调用链路。但我不会直接断定库存服务业务代码慢，而是继续看库存服务自身 trace、P95/P99、错误率、连接池等待、网络耗时、超时和重试情况，确认慢点是在调用方、传输过程、库存服务入口，还是库存服务内部依赖。

重试要谨慎使用。它可以解决偶发网络抖动或临时错误，但如果下游本来就慢或已经过载，重试会增加下游 QPS 和连接占用，把局部故障放大成级联故障。因此我会区分错误类型，只对临时错误做有限次数重试，并配合短超时、指数退避、jitter、熔断和幂等设计。对于下单、支付、扣库存这类非天然幂等操作，还必须先设计幂等 key，避免重试造成重复写入。

如果接口平均延迟是 70ms，但 P99 是 2s，并且用户反馈偶尔很慢，我会重点看 P99 和慢请求 trace。因为平均延迟会掩盖少量长尾请求，而 P99 更能反映最慢 1% 用户的真实体验。接下来我会拆这些慢请求的耗时，看是否存在连接池等待、锁等待、缓存 miss、下游 RPC 超时、GC 暂停或重试放大等问题。

如果 P99 突然升高但平均延迟变化不大，我会判断这更像是少部分请求出现长尾延迟，而不是所有请求整体变慢。下一步会抽取 P99 慢请求的 trace，按应用、缓存、数据库、下游 RPC 和资源等待拆分耗时，重点看缓存 miss、热点、锁等待、连接池排队、下游超时和重试等偶发因素。

拿到一条 P99 慢请求 trace 后，我会先看每个 span 的耗时分布，找出最长的阶段，而不是只看接口总耗时。比如是应用计算、Redis、MySQL、下游 RPC、连接池等待还是锁等待。找到最长阶段后再继续下钻，区分是真正执行慢，还是在等待资源、网络传输、下游排队或重试放大。

## HTTP 超时与重试

### 当前进度

- [HTTP 超时与重试](./fundamentals/http-timeout-retry.md)

### 本节核心理解

超时负责“别无限等”，重试负责“临时失败再试一下”，幂等负责“重试不会造成重复副作用”。没有超时，慢下游会长期占用调用方线程、协程、连接和队列资源；没有边界的重试，会在下游变慢时把局部故障放大成全链路故障。

### 短回答背后的解释

#### 为什么没有超时会拖住调用方资源

Product API 调用 Inventory API 时，如果没有超时，调用方会一直等待下游返回。等待期间，请求并不是“什么都不消耗”，它会占住 Product Service 的处理资源，例如线程或协程、HTTP client 连接、请求上下文、内存和队列位置。上游客户端也拿不到响应，只能继续等待。如果这种请求变多，Product Service 自己的资源会被慢下游拖满，即使其他不依赖库存服务的接口也可能被影响。

#### 为什么不同阶段需要不同 timeout

一次 HTTP 调用不是一个单一动作，而是由多个阶段组成：先从连接池拿连接；如果没有空闲连接，可能要 DNS 解析、建立 TCP 连接、做 TLS 握手；请求发出去后，还要等待下游处理并返回响应。每个阶段卡住的原因不同，应该有不同边界。连接池等待超时保护调用方本地资源，connect timeout 处理下游不可达，TLS handshake timeout 处理握手卡住，read timeout 处理下游响应慢，total request timeout 保证整次调用不超过入口预算。只设置一个笼统超时，很容易让某个阶段等待过长。

#### 为什么下游 timeout 不能等于入口总预算

入口 API 的 500ms 是整条请求链路的总预算，不只是库存服务的预算。这个时间里还包括网关处理、应用计算、Redis 或 MySQL、库存 RPC、响应编码、日志指标 trace，以及必要的 buffer。如果库存服务 read timeout 也设置成 500ms，一旦库存服务真的等满 500ms，入口请求基本已经没有时间完成后续处理，甚至可能入口已经超时但下游还在执行。合理做法是把入口预算拆成子预算，例如库存服务只给 100ms 到 150ms，并为其他阶段和降级留空间。

#### 为什么 400 和 401 不应该重试

重试只适合“可能恢复”的临时故障，而 400 和 401 通常不是这类问题。400 表示请求参数、格式或语义有问题，同样的请求重复发送，参数不会自己变正确。401 表示未认证或认证信息无效，同样的请求重复发送，权限也不会自己恢复。对这类错误重试，只会增加调用方和下游的流量、日志噪声和排查成本。正确做法是快速失败，把错误暴露给调用方或修正请求/认证逻辑。

#### 为什么 503 可以重试但不能立刻重试

503 Service Unavailable 通常表示服务暂时不可用，例如实例发布、短暂过载、依赖抖动或负载均衡摘除过程中的短暂失败。这类错误有恢复可能，所以可以有限重试。但如果所有调用方收到 503 后立刻重试，会把原始流量和重试流量同时打回正在恢复的下游，可能形成重试风暴。backoff 的作用是让重试间隔逐渐变长，给下游恢复时间；jitter 的作用是加入随机抖动，避免大量请求在同一时刻再次打过去。

#### backoff 和 jitter 的区别

backoff 解决的是“不要马上重试”的问题：第一次失败后等 50ms，第二次等 100ms，第三次等 200ms，让重试间隔逐渐变长，给下游恢复时间。jitter 解决的是“不要大家同时重试”的问题：在 backoff 的基础上加入随机抖动，例如 100ms 变成 100ms 到 130ms 之间的随机值。这样即使很多请求同一时间失败，它们的下一次重试也会被打散，而不是在同一个时间点再次冲击下游。

#### 为什么创建订单超时后不能直接重试

POST /orders 是有副作用的写操作，请求超时并不代表服务端一定没有执行成功。可能服务端已经创建了订单，只是响应在返回途中超时；也可能服务端还在处理中；也可能确实失败了。如果客户端或服务端直接用同一个业务动作再发一次，而后端没有幂等 key、唯一约束或请求去重机制，就可能创建两笔订单。正确做法是让客户端或调用方携带稳定的幂等 key，例如 order_request_id，后端用这个 key 保证同一个业务请求只产生一次效果；重试时如果发现已经处理过，就返回同一笔订单结果。

#### 写请求应该怎样设计重试策略

读请求通常没有副作用，重试失败的 GET 更多是在重复读取数据，所以可以对连接错误、短暂超时、429、503 做有限重试，并配合 backoff、jitter 和重试预算。写请求不同，它会改变状态，例如创建订单、扣库存、扣款、发券。写请求的安全策略不是“绝不重试”，而是“先让重试安全”。常见做法是调用方生成稳定幂等 key，后端用唯一索引或幂等表记录处理状态；如果第一次已经成功，重试返回同一结果；如果正在处理中，返回处理中或让调用方查询结果；如果明确失败，才允许重新发起或进入补偿流程。对于支付、扣款这类强副作用操作，通常还要配合状态机、回调幂等和对账补偿。

#### 为什么同一个 Idempotency-Key 要返回同一笔订单

Idempotency-Key 表示“这是同一次业务意图”。客户端第一次下单超时后重试，并不代表用户想买两次，而是想确认刚才那次下单有没有成功。后端发现同一个 key 已经创建过订单时，应该返回之前的订单结果，而不是再创建一笔。这样无论调用方重试几次，最终业务效果都是一笔订单。工程上通常会给幂等 key 建唯一索引，记录处理状态和关联订单 id，避免并发重试时穿透幂等保护。

#### 第一次请求处理中时，第二次重试怎么处理

同一个 Idempotency-Key 的第一次请求还没处理完，第二次请求已经来了，这时不能让两个请求同时创建订单。常见做法是幂等表记录状态：PROCESSING、SUCCESS、FAILED。第二次请求看到 PROCESSING，可以直接返回 202 Accepted 或 409 Conflict，并提示稍后查询；也可以在服务端短暂等待第一次请求完成，然后返回同一结果。选择哪种取决于接口体验和系统复杂度，但核心原则是不并发执行两次写操作。处理中状态还需要过期和恢复机制，避免第一次请求崩溃后 key 永远卡住。

#### 为什么多层重试会乘法放大 QPS

如果入口是 1000 QPS，每个请求对下游最多尝试 3 次，那么下游最坏会看到 3000 QPS。更危险的是多层服务都重试：A 调 B 最多 3 次，B 调 C 也最多 3 次，那么 C 最坏会看到 1000 * 3 * 3 = 9000 QPS。这说明重试不是局部行为，它会改变整条链路的流量形态。服务链路越深，每层都独立重试越危险，所以需要统一重试策略、重试预算、熔断和限流，避免下游在故障时被重试流量压垮。

#### 重试预算限制的是什么

重试预算限制的是额外补发的重试流量，而不是原始请求流量。比如原始流量是 1000 QPS，重试预算是 10%，就表示最多允许 100 QPS 的重试流量。超过预算后，即使还有请求失败，也应该快速失败、降级或交给熔断器处理，而不是继续补发。这样做的目标是让重试只处理少量临时故障，而不是在下游持续异常时制造重试风暴。

#### 重试预算怎么实现

重试预算通常按下游服务、接口或调用方维度统计。简单实现是滚动窗口计数：在最近 10 秒或 1 分钟内记录 original_count 和 retry_count，如果 retry_count 小于 original_count * budget_ratio，就允许重试，否则快速失败或降级。另一种实现是 token bucket：每个原始请求按比例生成少量 retry token，例如 10% 预算就是每个原始请求生成 0.1 个 token；每次重试消耗 1 个 token，token 不足就不重试。token bucket 可以设置容量上限，允许少量突发，但不会无限积累。工程上可以本地每实例近似实现，也可以用 Redis/sidecar 做全局预算；很多场景本地近似已经足够，因为目标是限制重试风暴，而不是做精确计费。

#### 为什么弱依赖要短 timeout 和可降级

商品详情页的核心目标是让用户看到商品基本信息并能继续浏览或购买，精确库存数量通常只是辅助信息。如果库存服务变慢，却让整个商品详情接口等到超时，等于让弱依赖拖垮主链路。后端设计要区分强依赖和弱依赖：强依赖失败可能必须失败返回，弱依赖失败则应该短 timeout、少重试、可降级。库存精确数量可以用 Redis 快照、模糊文案“库存紧张/有货”或隐藏字段替代，这样用户仍然能看到商品详情，系统也能保护核心链路。

### 本节面试表达补充

如果 Product API 调用库存服务没有设置超时，而库存服务卡住，Product API 会一直等待下游返回。这个等待会占住调用方的线程/协程、HTTP 连接、请求上下文和队列资源，也无法及时返回给上游。并发一高，慢下游就可能把调用方资源拖满，所以跨服务调用必须设置明确的超时边界。

HTTP 调用通常不只需要一个总超时，而要按阶段设置边界。比如连接池等待超时、DNS/connect timeout、TLS handshake timeout、read timeout 和 total request timeout。因为这些阶段代表不同问题：可能是本地连接池满了，可能是下游不可达，也可能是下游处理慢。分开设置才能快速定位和保护资源。

入口 API 的总预算要拆给各个阶段，不能让某个下游调用吃完整个预算。比如入口总预算是 500ms，库存服务 read timeout 不应该也设置成 500ms，而应该给库存调用一个更小的子预算，同时给应用计算、缓存/数据库、响应返回、日志观测和 buffer 留时间。否则下游一旦等满，入口就会跟着超时，系统也很难优雅降级。

我不会对 400 或 401 这类错误做重试。400 通常代表请求参数或格式错误，401 代表认证失败，同样的请求重复发送也不会变成功。重试应该用于临时故障，例如连接 reset、短暂网络错误、429 或 503，而不是用于确定性的客户端错误或权限错误。

503 通常代表服务暂时不可用，可以做有限重试，但不能立刻无脑重试。因为下游可能正在过载或发布恢复中，立刻重试会把更多流量打过去，导致重试风暴。更合理的做法是配合最大重试次数、backoff、jitter、重试预算和熔断，必要时快速失败或降级。

backoff 是让重试间隔逐渐变长，避免失败后立刻反复打下游；jitter 是在等待时间里加入随机抖动，把大量调用方的重试时间打散，避免它们在同一时刻再次请求下游。两者通常配合使用：backoff 降低重试频率，jitter 避免重试流量同步形成尖峰。

POST /orders 这类创建订单的写操作不能因为超时就直接重试，必须先看后端是否支持幂等。因为超时只代表调用方没有及时拿到结果，不代表服务端没有创建成功。安全做法是由调用方传入稳定的幂等 key，后端基于幂等 key 或唯一约束做去重，保证同一次业务请求重复提交时只创建一笔订单，并返回一致结果。

读请求和写请求最大的区别是副作用。GET /products/1001 这类读请求通常可以对临时错误做有限重试，并配合 backoff、jitter 和重试预算；POST /orders 这类写请求不能无脑重试，必须先设计幂等 key、唯一约束或幂等表，保证重复提交只产生一次业务效果。对于超时后的不确定状态，优先通过幂等 key 查询已有结果，或进入状态机和补偿流程。

如果客户端用同一个 Idempotency-Key 重试下单请求，后端发现这个 key 已经对应一笔订单，应该返回之前那笔订单或同一处理结果，而不是新建订单。因为幂等 key 代表同一次业务意图，重复请求只是在确认结果，不应该产生新的副作用。

如果同一个 Idempotency-Key 的第一次请求还在处理中，第二次重试来了，后端可以返回处理中、409/202，或者短暂等待第一次请求完成后返回同一结果。无论选择哪种方式，关键是不能让两个请求同时执行创建订单这类写操作，否则幂等保护会失效。

重试会放大下游流量。入口 1000 QPS，如果每个请求最多 1 次原始调用加 2 次重试，下游最坏会看到 3000 QPS。如果调用链上多层服务都重试，放大是乘法关系，例如 A 调 B 最多 3 次、B 调 C 最多 3 次，C 最坏会看到 9000 QPS。因此重试必须有限次数，并配合重试预算、限流和熔断。

重试预算用来限制额外重试流量。例如原始流量是 1000 QPS，重试预算是 10%，最多只允许 100 QPS 的重试。超过预算后应该快速失败、降级或熔断，避免在下游已经异常时继续用重试流量加压。

重试预算可以用滚动窗口计数或 token bucket 实现。滚动窗口里统计某个下游最近一段时间的原始请求数和重试请求数，只有 retry_count 小于 original_count * budget_ratio 才允许继续重试。token bucket 则是原始请求按比例产生 retry token，每次重试消耗 token，token 不足就快速失败或降级。实际工程里通常按下游服务和接口维度做预算，并和超时、backoff、jitter、熔断一起使用。

库存精确数量在商品详情里通常是弱依赖，不应该拖垮主链路。因此 Product API 会给库存 RPC 设置更短 timeout，重试次数也更保守；如果库存服务失败，就用 Redis 库存快照、模糊库存文案或隐藏精确库存做降级。技术策略要服务业务优先级，非核心信息要允许降级。

## 连接池

### 当前进度

- [连接池](./fundamentals/connection-pool.md)

### 本节核心理解

连接池是一组可复用的网络连接，用来减少频繁建连成本，并限制应用同时打到数据库、Redis 或下游服务的并发。连接池不是越大越好：太小会导致请求等待连接，太大会把下游连接数、并发查询和资源调度压力打满。

### 短回答背后的解释

#### 为什么每次请求都新建连接很贵

数据库连接不是普通函数调用，它背后有网络连接、认证、协议握手，有些场景还包括 TLS。每次请求都新建连接，会让调用方和数据库反复做建连和释放工作。低 QPS 时可能只是多一点延迟，高 QPS 时会产生大量短连接，数据库要花资源处理连接建立、认证、内存分配和关闭，而不是专心执行查询。连接池通过复用已有连接，把这部分固定成本摊薄，同时也限制了同时进入数据库的并发数量。

#### 为什么连接池不是越大越好

连接池不仅是复用连接的工具，也是控制下游并发的闸门。数据库能高效并行处理的查询数量是有限的，超过这个范围后，并发更多不一定更快，反而会增加上下文切换、锁竞争、内存占用、磁盘 IO 排队和查询等待。如果数据库稳定能承受 100 个并发查询，应用把连接池开到 200，并且瞬时真的放进 200 个查询，数据库可能整体变慢，所有请求 P99 都升高。连接池调大只能减少应用侧等待，不会凭空增加数据库处理能力。

#### 为什么池等待高不能立刻大幅调池

等待数据库连接 1200ms、SQL 查询 80ms，说明当前慢点发生在借连接阶段，而不是这条 SQL 的执行阶段。但这不自动等于“把池调大就能解决”。要先看连接池 active connections 是否长期等于 max、pending requests 是否升高、connection acquire time 是否变长；再看数据库 CPU、锁等待、慢 SQL、事务耗时和最大连接数。如果连接被长事务、慢查询或泄漏占住，单纯调大池会让更多请求进入数据库，可能把数据库拖慢。合理做法是先定位连接为何不释放，再在数据库容量允许时小步调大，并继续观察 P95/P99、DB CPU 和 pool wait。

#### 为什么看到 connection timeout 要先澄清含义

日志里的 connection timeout 可能指不同问题：等待连接池里的空闲连接超时，说明本地池可能耗尽；建立数据库或 HTTP 连接超时，说明下游不可达、网络慢或连接建立失败；也可能是客户端库把多种 timeout 统一包装成 connection timeout。排查时要先看错误来源、异常类型和 trace span，确认是 pool wait timeout、connect timeout 还是 read/query timeout。只有确认是池等待超时后，才按连接池耗尽方向继续看 active、idle、pending、acquire time、慢 SQL、长事务、连接泄漏、流量突增和数据库容量。

#### active 打满但 DB CPU 不高说明什么

active connections 长期等于 max、pending requests 高，说明应用侧连接池被占满，新请求在等连接。但数据库 CPU 不高，说明瓶颈不一定是数据库计算能力耗尽。可能是连接被代码借走后没有归还，也可能是长事务或慢 SQL 持有连接很久但并不消耗太多 CPU，例如锁等待、IO 等待、深分页扫描、外部依赖等待期间仍占着事务连接。也可能只是流量突增或池配置过小。排查时要看连接持有时间、事务耗时、慢 SQL、lock wait、连接泄漏检测日志和应用代码的 release/finally 路径。

#### 为什么异常路径必须归还连接

连接池的前提是“借了会还”。如果正常路径归还连接，但异常路径忘记 release，连接池会出现连接泄漏。泄漏不是一次性报错，而是可用连接逐渐减少：一开始只是偶发等待，随后 active connections 长期接近 max、idle 变少、pending requests 增加，最后大量请求等待连接超时。因为数据库 CPU 可能并不高，这类问题容易被误判成连接池太小。所有语言都要用 try/finally、try-with-resources、defer Close 或 context manager 保证连接一定归还。

#### 为什么慢 SQL 会打满连接池

连接池能承载多少请求，不只取决于连接数量，还取决于每个请求持有连接多久。如果 SQL 从 50ms 变成 800ms，单个连接的周转速度会明显下降。请求进入速度不变，但连接归还速度变慢，active connections 会逐渐升高并达到 max，新请求只能排队等待连接。于是一个慢 SQL 不仅让自己的查询变慢，还会让其他请求在连接池前等待，最终表现为 API P99 升高和 connection acquire time 增加。

#### 为什么池等待超时要小于入口预算

入口 API 的 500ms 是整次请求的总预算，不只是等待数据库连接的预算。如果连接池等待超时设置成 1s，请求可能在池前排队 1 秒才失败或拿到连接，但此时上游用户请求早就超过 500ms 预算了。即使最终拿到连接，也已经没有时间执行 SQL、编码响应和记录观测数据。池等待超时应该是入口预算中的一个子预算，例如几十到一两百毫秒，池满时快速失败、降级或限流，而不是排队到入口超时。

#### 如何按实例数估算连接池大小

连接池大小不能只看单个实例，要看所有实例加起来会给数据库多少连接。如果数据库最多希望承受 200 个应用连接，Product API 有 10 个实例，平均每个实例就是 20 个连接。实际配置通常还要给其他服务、后台任务、运维连接、连接波动和扩容留余量，所以每实例可能控制在 15 到 20 左右。不能每个实例都配置 30，因为 10 个实例就是 300 个连接，已经超过数据库目标容量。这个估算只是起点，最终还要结合压测、DB CPU、IO、锁等待、连接等待和查询耗时调整。

#### 为什么深分页会拖垮连接池

`LIMIT 20 OFFSET 100000` 这类深分页查询通常需要数据库扫描或跳过大量行，再返回后面的少量结果。即使最终只返回 20 行，数据库也可能做大量排序、扫描和 IO。查询变慢后，连接被持有的时间变长，连接归还速度下降，active connections 更容易打满，新请求开始 pending。此时连接池等待高是结果，不是根因。正确方向是优化分页方式，例如使用合适索引、基于游标/created_at + id 翻页、限制最大页数，必要时做搜索/归档方案，而不是直接把连接池调大让更多慢查询同时进入数据库。

### 本节面试表达补充

#### 连接池为什么不是越大越好

连接池大小要看数据库本身的承受能力，连接池不仅是复用连接的工具，也是对数据库的一种保护。池太小会让应用侧请求等待连接，导致 P99 升高；但池太大也不是好事，因为它会允许更多查询同时进入数据库，超过数据库能高效处理的范围后，会带来上下文切换、锁竞争、内存占用、IO 排队和查询等待。连接池大小应该从数据库容量、服务实例数和接口并发倒推，并通过压测和监控逐步调整。

当前自己的简短回答：要看数据库本身的承受能力，连接池的大小，是对数据库的一种保护。

如果每次请求都新建数据库连接，用完就断开，会有明显的建连和释放成本，包括 TCP/TLS、认证和数据库协议握手。高并发下，大量短连接会增加数据库和应用的资源消耗，也会拉高请求延迟。连接池通过复用连接减少建连开销，同时通过最大连接数控制打到数据库的并发。

连接池不是越大越好。它同时承担复用连接和限制下游并发的作用。池太小会让请求等待连接，池太大会让过多查询同时进入数据库，超过数据库能高效处理的范围，导致上下文切换、锁竞争、内存和 IO 压力上升。连接池大小应该从数据库容量、服务实例数和接口并发倒推，而不是单纯调大。

如果 trace 显示等待数据库连接 1200ms、SQL 查询 80ms，我会判断瓶颈主要在连接池等待，而不是这条 SQL 本身。但我不会直接把连接池从 16 调到 100，而是先看 active、idle、pending、connection acquire time、连接泄漏、长事务、慢查询和数据库容量。如果确认数据库还有余量且只是池配置偏小，可以小步调大连接池；如果是连接被慢事务或泄漏占住，就要先修复根因。

看到大量 connection timeout 时，我不会立刻判断连接池太小，而是先确认 timeout 发生在哪个阶段：是等待池连接超时、建连超时，还是查询/read 超时。确认是池等待超时后，再看 active connections、pending requests、connection acquire time、连接泄漏、长事务、慢 SQL、流量突增和数据库容量。只有确认数据库还有余量且池配置偏小，才会小步调大连接池。

如果连接池 active 长期等于 max、pending 很高，但数据库 CPU 不高，我会怀疑连接被长时间占用而不是数据库算力打满。常见原因包括连接泄漏、长事务、慢 SQL 或锁等待、流量突增、池配置偏小，以及应用在持有连接期间做了不必要的外部等待。下一步会看连接持有时间、事务耗时、慢 SQL、lock wait、泄漏检测日志和代码是否可靠 release。

借到连接后必须用 finally release、defer Close 或类似机制保证归还。否则异常路径一旦忘记释放，就会发生连接泄漏，池里的可用连接越来越少，最终 active 长期打满、pending 请求增多，大量请求在等待连接时超时。连接泄漏通常要通过连接持有时间、泄漏检测日志和代码审查来定位。

慢 SQL 会导致连接池被打满，因为连接被持有的时间变长，归还速度下降。即使请求 QPS 不变，原来每个连接 50ms 就能归还，现在要 800ms 才归还，连接周转率大幅降低，active connections 更容易达到 max，新请求就会在池前排队，API P99 也会升高。

连接池等待超时必须小于入口 API 的总预算。如果入口只有 500ms，而池等待超时设成 1s，请求可能还在等连接时，用户侧已经超时了。合理做法是把池等待作为总预算的一部分，设置更短的等待上限，并给真正的 SQL 查询、响应返回和日志观测留时间。

估算连接池大小时，要用数据库总连接预算除以服务实例数，并预留余量。比如数据库最多希望承受 200 个应用连接，Product API 有 10 个实例，粗略平均是每实例 20 个连接；实际可能配置 15 到 20，给其他服务、运维连接和扩容留空间。不能只看单实例觉得 30 不大，因为 10 个实例合起来就是 300 个连接。

如果连接池 active 和 pending 都很高，同时慢 SQL 里出现大量深分页查询，我会优先怀疑 SQL/分页方式导致连接持有时间变长，而不是直接判断连接池太小。深分页会让数据库扫描或跳过大量行，查询变慢后连接归还变慢，进而造成池等待。修复方向是优化索引、改成游标分页、限制深分页或引入更适合查询的检索方案。
