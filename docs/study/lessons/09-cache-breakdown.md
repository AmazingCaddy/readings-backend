# 第九课：Redis 缓存击穿

原文：[Redis 缓存击穿](../../cache/cache-breakdown.md)

## 核心理解

缓存击穿是指某个热点 key 在过期或被删除的瞬间，大量请求同时发现缓存 miss，然后一起回源数据库。它和缓存雪崩不同：击穿通常集中在单个热点 key；它和缓存穿透也不同：击穿访问的数据通常真实存在，只是缓存刚好失效。治理目标不是让缓存永远不 miss，而是让回源并发变少、可控、可降级。

## 短回答背后的解释

### 为什么热点 key 过期会击穿数据库

Cache-Aside 里缓存 miss 后回源数据库是正常流程。普通 key 的 miss 通常流量不大，数据库查一次或少量几次就能承受。但热点 key 不一样：如果 `product:1001` 每秒有 1 万次请求，缓存过期的瞬间，大量请求都会发现 Redis 没有值。如果没有互斥回源、逻辑过期或限流保护，这些请求会同时查询数据库。数据库本来只需要查一次来重建缓存，却被重复查询打满，连接池、CPU、IO 和 API P99 都会恶化。

### 互斥回源怎么保护数据库

互斥回源的目标是把同一个热点 key 的并发回源压缩到一个或少数几个请求。请求发现 Redis miss 后，先尝试在 Redis 里创建一个短 TTL 的锁，例如 `SET lock:product:1001 requestId NX EX 5`。拿到锁的请求负责查数据库、重建缓存、释放锁；没拿到锁的请求不应该直接查数据库，而是短暂等待几十毫秒后再读缓存，或者返回旧值、降级结果、快速失败。锁必须有过期时间，避免持锁请求崩溃后锁永远不释放；释放锁时要校验 value 是自己的 requestId，避免误删其他请求后来拿到的锁。

### 为什么释放锁要校验 requestId

互斥回源锁有过期时间。假设请求 A 拿到锁后查数据库，但数据库很慢，锁先过期了；请求 B 随后拿到了新的锁并开始重建缓存。此时请求 A 终于返回，如果它直接执行 `DEL lock:product:1001`，就会把请求 B 的锁删掉。其他请求看到锁不存在，又可能继续抢锁回源，互斥保护被破坏。所以释放锁必须先判断锁的 value 是否等于自己的 `requestId`，只有属于自己的锁才能删除。这个判断和删除要保证原子性，通常用 Lua 脚本实现 compare-and-delete。

### 没拿到锁的请求为什么不能一直等

互斥回源保护的是数据库，但不能把等待压力无限转移到应用层。没拿到锁的请求如果一直阻塞，会占住 HTTP 连接、线程/协程、请求上下文和上游等待预算；并发一高，Product API 自己也会被拖满。更合理的做法是给等待设置很小的预算，例如 sleep 20 到 50 ms 后再读一次缓存，最多尝试一两次。如果缓存已经重建，直接返回；如果仍然没有，就根据业务返回旧值、降级文案、快速失败或让客户端稍后重试。

### 逻辑过期的好处和代价

逻辑过期是指 Redis key 物理上不立即删除，value 里额外保存一个 `expireAt`。请求读到 value 后，先判断逻辑过期时间：如果没过期，直接返回；如果已经过期，也可以先返回旧值，同时只让一个后台刷新任务或拿到刷新锁的请求去查数据库并更新缓存。它的好处是热点 key 永远有值可返回，避免大量请求同时等待互斥锁或同时回源数据库，接口 P99 更稳定。代价是牺牲数据新鲜度：用户可能短时间看到旧商品详情；如果后台刷新失败，旧值窗口会变长。因此逻辑过期适合允许短暂旧值的热点读场景，不适合余额、权限、支付状态这类强一致数据。

### 热点 key 更新后为什么不宜直接删除缓存

普通 Cache-Aside 写路径常用“更新数据库后删除缓存”，让下一次读 miss 后重新加载。但如果这个 key 是热点 key，直接删除会让缓存瞬间变空，大量读请求会同时 miss；如果没有互斥回源或逻辑过期保护，就会形成缓存击穿。热点 key 的写路径通常要更谨慎：可以在数据库更新成功后主动重建缓存，用新值覆盖旧值；也可以使用逻辑过期，让旧值继续可读并异步刷新；如果仍然采用删除缓存，就必须配合互斥回源、回源限流和降级，确保只有少量请求能访问数据库。

### 超热点低频配置适合组合策略

首页「今日爆款商品」这类配置 QPS 很高，但修改频率低，并且允许用户短时间看到旧配置。它更适合逻辑过期和写后主动刷新的组合：读路径上 Redis 物理 key 一直存在，即使逻辑过期也可以先返回旧值，保证接口 P99 稳定；写路径上运营修改配置并提交数据库后，主动刷新 Redis，把旧值窗口缩短到很小。这样既避免了物理删除带来的击穿窗口，也避免只靠自然 TTL 导致旧配置持续太久。

### 互斥锁 TTL 怎么取舍

互斥回源锁的 TTL 不能随便设。TTL 太短时，第一个请求还没查完数据库、还没写回缓存，锁就已经过期，其他请求又能抢到锁并开始回源，互斥效果被削弱。TTL 太长时，如果持锁请求卡住、崩溃或重建耗时异常，其他请求会长时间拿不到锁，只能等待到预算耗尽后失败或降级。实际设置通常要略大于正常缓存重建耗时的 P99，并配合短等待、重读缓存、降级和告警。对于偶发 2 秒这种长尾重建，要先看是否能优化回源 SQL 或改成逻辑过期，而不是单纯把锁 TTL 拉得很长。

### 如何用指标确认缓存击穿

缓存击穿是一个短时间窗口内的尖峰问题，排查时要按时间线把 Redis、DB 和 API 指标对齐。Redis 层看热点 key 的 hit/miss、key 是否过期、miss 是否集中在同一时间、互斥锁获取成功/失败次数、缓存重建次数和 Redis 延迟。DB 层看回源 QPS 是否瞬间升高、连接池等待、SQL P95/P99、锁等待、CPU/IO 是否被打高。API 层看接口 P95/P99、错误率、等待互斥锁耗时、降级次数、快速失败次数和上游重试量。只有这些指标同一时间抖动，才能确认不是普通慢 SQL，而是热点 key miss 导致的并发回源。

### 击穿、穿透、雪崩怎么区分

缓存击穿关注单个热点 key：数据通常存在，但热点 key 过期或被删除，导致大量请求同时 miss 并回源数据库。缓存穿透关注不存在的数据：请求的 key 在缓存没有，数据库也没有，如果没有参数校验、空值缓存或布隆过滤器，每次都会绕过缓存打到数据库。缓存雪崩关注大面积失效：大量 key 同时过期，或者 Redis 整体不可用，导致原本由缓存承载的流量大面积转移到数据库。可以简单记：击穿是“一个热 key”，穿透是“不存在”，雪崩是“一大片”。

## 面试表达

缓存击穿发生在热点 key 过期或被删除的瞬间。普通缓存 miss 是正常回源，流量通常比较小；但热点 key 一旦 miss，大量请求会同时回源数据库，数据库承受很多重复查询，可能连接池打满、P99 飙升，甚至触发上游重试。它的核心风险不是 miss 本身，而是热点 key 的并发回源失控。

互斥回源的核心是同一个 key 缓存 miss 时，只允许一个请求回源数据库并重建缓存。通常用 Redis `SET NX EX` 做短 TTL 锁，拿到锁的请求查 DB、写缓存、释放锁；没拿到锁的请求短暂等待后重读缓存，或者返回旧值/降级。这样数据库只承担一次或少量回源，而不是被所有并发请求同时打穿。

释放互斥锁时不能简单 `DEL lock`，因为锁可能已经过期并被其他请求重新获取。旧请求回来后如果直接删除，就会误删别人的锁，导致更多请求同时回源。正确做法是锁 value 放一个唯一 `requestId`，释放时只有 value 等于自己的 requestId 才删除，并用 Lua 保证检查和删除是原子操作。

没拿到互斥锁的请求不能一直等。等待也会消耗 API 线程、HTTP 连接和上游请求预算，所以应该短暂等待后重读缓存；如果仍然没有缓存，就返回旧值、降级结果、快速失败或提示稍后重试。互斥回源的目标是让数据库回源可控，同时让应用层等待也有边界。

逻辑过期的思路是缓存物理上不过期，value 里保存逻辑过期时间。热点 key 逻辑过期后，请求可以先返回旧值，同时异步触发一个刷新任务重建缓存。它的好处是用户请求不必都等待回源，数据库压力也更平滑；代价是短时间数据不够新鲜，后台刷新失败时旧值会持续更久，所以要有监控、刷新重试和兜底策略。

对于热点 key，更新数据库后直接删除 Redis key 会制造缓存空窗，下一波读请求可能同时 miss 并打到数据库。所以热点 key 通常不会只依赖“删缓存后下次读重建”。更稳的做法是写后主动刷新缓存、使用逻辑过期保留旧值并异步刷新，或者至少在删除后用互斥回源保护数据库。

对于首页爆款配置、热门榜单这类超热点低频更新数据，我会组合使用逻辑过期和写后主动刷新。逻辑过期让读请求始终有旧值可返回，避免热点 key 过期时击穿数据库；写后主动刷新则在运营修改后尽快把新值写入 Redis，缩短用户看到旧值的时间。

互斥回源锁 TTL 要略大于正常缓存重建耗时的 P99。太短会导致第一个请求还没重建好，锁已经过期，其他请求继续抢锁回源；太长则会在持锁请求异常或慢查询时，让其他请求长时间等待并最终失败。锁 TTL、等待预算、降级策略和回源耗时监控要一起设计。

确认缓存击穿时，我会把 Redis、DB、API 三层指标按同一个时间窗口对齐。Redis 看热点 key 的 miss 是否突然升高、锁竞争和重建次数；DB 看回源 QPS、连接池等待、SQL P99 和资源压力是否同时升高；API 看 P99、错误率、等待锁耗时和降级量。缓存击穿通常表现为 Redis miss 尖峰、DB 回源尖峰和 API P99 尖峰同时出现。

缓存击穿、穿透、雪崩的核心区别是范围和数据是否存在。击穿是单个热点 key 失效，大量请求同时回源；穿透是请求不存在的数据，缓存和数据库都没有；雪崩是大量 key 同时失效或 Redis 整体不可用，导致大面积流量打到数据库。
