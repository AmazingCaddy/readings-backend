# 第五课：事务隔离级别

原文：[事务隔离级别](../../database/transaction-isolation.md)

## 核心理解

事务不是自动解决所有并发问题。隔离级别决定多个事务同时执行时彼此能看到什么中间状态，但库存、余额、支付、订单状态这类强一致业务，还需要条件更新、唯一索引、锁、短事务和业务状态机一起兜底。

## 短回答背后的解释

### 为什么先读后写会导致超卖

库存只剩 1 件时，两个事务几乎同时执行 `SELECT stock`，都可能读到 `stock = 1`。应用层都判断库存充足，于是两个事务都继续执行扣减。如果更新语句没有把 `stock > 0` 作为写入条件，或者业务没有正确处理并发冲突，就可能两个订单都成功，库存被扣成 -1，或者库存最终看起来是 0 但实际卖出了两件。这类问题的根因是读和写之间存在并发窗口。

### 为什么条件更新更安全

`UPDATE inventory SET stock = stock - 1 WHERE sku_id = ? AND stock > 0` 把“判断库存是否大于 0”和“扣减库存”合并成同一条写语句。数据库执行这条语句时会在写入阶段判断条件是否仍然成立，并对同一行的并发更新做冲突控制。相比先 SELECT 再 UPDATE，条件更新没有把业务判断暴露在应用层的并发窗口里。结果可以通过 affected rows 判断：1 表示扣减成功，0 表示库存不足或记录不存在。

### affected rows = 0 怎么理解

`affected rows = 0` 表示这条 UPDATE 没有更新任何行。在 `WHERE sku_id = ? AND stock > 0` 的扣库存场景里，常见原因是库存已经不足、sku 记录不存在，或者条件不再成立。业务上应该把它当作扣减失败处理：回滚当前事务，不再插入订单，并返回售罄或失败。不能忽略 affected rows，也不能在扣减失败后继续创建订单，否则会产生库存和订单不一致。

### 为什么扣库存和插入订单要在同一事务里

扣库存和插入订单共同组成一次下单业务结果，必须保证原子性。如果扣库存成功后，插入订单失败但没有回滚，就会出现库存减少但没有订单；如果订单插入成功但库存没有扣减，就会出现订单和库存不一致。把两步放在同一个事务里，可以在任何一步失败时 rollback，让数据库恢复到下单前状态；只有两步都成功时才 commit。

### 为什么防重复下单需要唯一索引兜底

应用层先查订单是否存在，再决定是否插入，在高并发下也有并发窗口。两个请求可能同时执行查询，都发现没有订单，然后都插入成功。解决这类问题不能只靠应用层判断，而要在数据库层建立唯一约束，例如 `UNIQUE(user_id, activity_id)` 或 `UNIQUE(idempotency_key)`。这样即使两个请求同时插入，数据库也只允许一个成功，另一个会收到唯一键冲突，业务再把它处理成已下单或幂等返回。

### 脏读和幻读怎么区分

读到其他事务还没有 commit 的数据叫脏读。它危险在于这个数据后续可能 rollback，并没有真正生效。如果当前事务基于这个未提交数据做库存、余额或订单判断，就可能产生错误决策。幻读则是同一个事务里两次范围查询，第二次多出或少了符合条件的行。可以简单记：脏读是读到未提交数据，不可重复读是同一行两次读结果不同，幻读是同一范围两次查行数变了。

### 什么是不可重复读

不可重复读是同一个事务里两次读取同一行，结果不一样。典型场景是事务 A 第一次读到订单状态 `CREATED`，事务 B 把同一行更新为 `PAID` 并提交，事务 A 第二次读同一行时看到 `PAID`。它读到的是别人已经提交的数据，所以不是脏读；问题在于同一事务内部对同一行的观察不稳定。对于依赖状态检查的业务，要注意隔离级别、行锁或条件更新。

### 什么是幻读

幻读发生在范围查询上。同一个事务第一次执行 `SELECT * FROM orders WHERE amount > 100` 查到 10 条；另一个事务插入一条 `amount = 200` 的订单并提交；原事务第二次执行同样的范围查询，查到 11 条。这时多出来的行就像“幻影”一样出现，所以叫幻读。它和不可重复读的区别是：不可重复读关注同一行内容变了，幻读关注同一范围里的行集合变了。

### 为什么隔离级别不是越高越好

隔离级别越高，数据库对并发异常的约束越强，但并发成本也越高。Serializable 的效果接近事务串行执行，能避免更多异常，但也会增加锁等待、冲突、死锁或序列化失败，降低吞吐并拉高延迟。生产系统通常不会所有业务都直接使用最高隔离级别，而是选择合适隔离级别，再用条件更新、唯一索引、乐观锁/悲观锁、短事务和状态机保证业务正确性。

### 为什么事务要尽量短

事务里调用外部 HTTP、等待 MQ 或做慢计算，会让事务持续时间变长。事务越长，数据库连接、行锁、MVCC 版本和相关资源持有越久，其他事务越容易等待，也更容易出现死锁或锁等待超时。长事务还会降低连接周转率，导致连接池 active 打满、pending 上升，最终表现为接口 P99 升高。因此事务里只放必要的数据库读写，把外部调用放到事务外，或者用 Outbox 等模式异步处理。

### 死锁和序列化失败怎么处理

死锁和序列化失败通常属于并发冲突，不是业务参数错误，所以可以做有限重试。重试时要重新执行整个事务，而不是从事务中间某一步继续。常见策略是最多重试 1 到 3 次，配合短暂 backoff 和 jitter，并记录冲突日志和指标。如果超过次数仍失败，就返回失败或让调用方稍后再试，避免无限重试放大数据库压力。

数据库通常会自动检测死锁，并通过回滚其中一个事务来打破死锁，释放锁资源。但这不代表业务请求自动成功。被回滚的事务会收到死锁错误，应用需要捕获这个错误，根据业务是否安全来决定是否有限重试整个事务。

## 面试表达

先查库存再扣库存在高并发下是不安全的。两个事务可能同时读到库存为 1，然后都认为可以下单，最后导致超卖或库存为负。更安全的做法是把判断和扣减合并成一条条件更新，例如 `UPDATE inventory SET stock = stock - 1 WHERE sku_id = ? AND stock > 0`，再根据 affected rows 判断扣减是否成功。

条件更新比先查后改更安全，因为它把判断条件和写入动作合并到同一条 SQL，由数据库在写入时原子判断 `stock > 0` 是否仍然成立。应用不再依赖之前 SELECT 出来的旧库存，而是根据 UPDATE 的 affected rows 判断扣减是否成功。

在条件扣库存里，`affected rows = 0` 通常表示库存不足、sku 不存在，或者更新条件已经不成立。业务应该回滚事务并返回售罄/失败，不能继续创建订单。`affected rows = 1` 才表示扣减成功，可以继续执行后续订单写入。

扣库存和插入订单通常要放在同一个数据库事务里，因为它们共同构成一次下单的原子结果。要么库存扣减成功并且订单创建成功后一起提交，要么任何一步失败都回滚，避免出现扣了库存但没有订单，或者有订单但库存没有扣的中间状态。

防重复下单不能只靠应用层先查再插入，因为两个并发请求可能同时查不到订单，然后都插入成功。应该在数据库层加唯一约束，例如 `UNIQUE(user_id, activity_id)` 或 `UNIQUE(idempotency_key)`，让数据库兜底保证同一个业务维度只能插入一次。唯一键冲突时，业务可以返回已下单或复用之前的订单结果。

读到其他事务还没有 commit 的数据叫脏读。它危险在于这个数据后续可能 rollback，并没有真正生效。如果当前事务基于这个未提交数据做库存、余额或订单判断，就可能产生错误决策。幻读则是同一个事务里两次范围查询，第二次多出或少了符合条件的行。

不可重复读是同一个事务里两次读取同一行，结果发生变化。它读到的是其他事务已经提交的数据，所以不是脏读；风险在于当前事务如果依赖第一次读取的状态做后续判断，第二次读到的新状态可能让业务逻辑前后不一致。

幻读是同一个事务里两次执行同样的范围查询，第二次多出或少了符合条件的行。它和不可重复读的区别在于：不可重复读关注同一行的值变了，幻读关注同一范围里的行集合变了。

事务隔离级别不是越高越好。Serializable 正确性更强，但会降低并发性能，增加锁等待、冲突、死锁或序列化失败。生产系统通常会选择合适的隔离级别，再配合条件更新、唯一索引、乐观锁/悲观锁、短事务和状态机来保证业务正确性，而不是所有场景都直接使用最高隔离级别。

事务里不建议调用外部 HTTP、等待 MQ 或做慢计算，因为这些操作会拉长事务时间。事务越长，锁、连接和 MVCC 版本持有越久，其他请求越容易等待，也更容易出现死锁。数据库事务应该尽量短，只包含必要的数据库读写；外部调用应放在事务外，或通过 Outbox 等方式异步处理。

死锁和序列化失败通常属于并发冲突，可以做有限重试，但必须重试整个事务，而不是从中间某一步继续。数据库会检测死锁并回滚其中一个事务来打破死锁；应用要捕获这个错误，按业务安全性决定是否用最大次数、backoff 和 jitter 重试，超过次数则返回失败或让调用方稍后再试。

### 怎么防止库存超卖

不要先查询库存再更新库存，而是把库存判断和扣减合并到一条条件更新里：`UPDATE inventory SET stock = stock - 1 WHERE sku_id = ? AND stock > 0`。然后根据 affected rows 判断是否扣减成功。扣库存和插入订单要放在同一个事务里，扣减失败就 rollback 并返回售罄，扣减成功后再插入订单并 commit。同时，用户维度防重复下单要用唯一索引或幂等 key 兜底，事务要尽量短，死锁或序列化失败可以有限重试整个事务。

当前自己的简短回答：不要先查询再更新，查询和更新要在一个 SQL 里面做。
