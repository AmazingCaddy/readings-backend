# 第十课：缓存穿透

原文：[缓存穿透](../../cache/cache-penetration.md)

## 核心理解

缓存穿透是指请求的数据在缓存和数据库里都不存在，导致每次请求都 Redis miss，然后继续查数据库。它和缓存击穿不同：击穿访问的数据通常存在，只是热点 key 刚好失效；穿透访问的数据本来就不存在。治理目标是尽量在进入数据库前拦住无意义请求，或者把空结果短时间缓存起来。

## 短回答背后的解释

### 为什么不存在的数据会穿透缓存

Cache-Aside 里 Redis miss 后会回源数据库。如果请求的是不存在的商品 ID，Redis 没有缓存，数据库也查不到。没有空值缓存、参数校验或布隆过滤器时，下一次同样或类似的请求仍然会 Redis miss 并查数据库。高 QPS 的不存在查询会持续消耗数据库连接、CPU、IO 和接口处理资源。缓存穿透和缓存击穿的区别在于：击穿访问的数据通常真实存在，只是热点 key 刚好过期；穿透访问的数据本来就不存在，缓存和数据库都没有。

### 空值缓存适合什么场景

如果同一个不存在的商品 ID 被反复访问，例如 `product:999999999`，第一次 Redis miss 后查数据库发现不存在，可以把一个 null marker 写入 Redis，例如 `product:999999999 = __null__`，并设置较短 TTL。后续请求命中 null marker 时，应用直接返回 404 或 not found，不再访问数据库。TTL 不能太长，因为这个商品 ID 以后可能真的被创建出来；如果旧的 null marker 还在，用户会继续查不到真实数据。

### 随机不存在 ID 为什么不能只靠空值缓存

空值缓存依赖“同一个不存在 key 被重复访问”。如果攻击者随机生成 100 万个不同的不存在商品 ID，每个 ID 可能只访问一次，那么 null marker 很难被再次命中，数据库压力不一定降下来，Redis 反而会存入大量空值缓存，占用内存并污染 key 空间。这类场景要把防线前移：网关或应用层先做参数校验和频率限制；对可能存在的商品 ID 建布隆过滤器，快速判断“这个 ID 一定不存在”并直接拒绝；同时结合 IP、用户、设备、UA、租户等维度做风控和限流。

### Bloom Filter 可能存在后仍要处理 DB not found

Bloom Filter 说“可能存在”只表示这个 ID 没有被过滤掉，不代表数据库一定有这条记录。由于 Bloom Filter 有假阳性，不存在的 ID 也可能被判断为可能存在。请求继续查 Redis，如果 Redis miss，再查数据库仍然可能返回 not found。这时应该写入短 TTL 的 null marker，避免后续同一个 ID 再次穿过 Bloom Filter、Redis 和数据库。也就是说，Bloom Filter 负责拦截大量“一定不存在”的随机 ID，空值缓存负责兜住漏进来的重复不存在 ID。

## 面试表达

缓存穿透是请求的数据在缓存和数据库里都不存在，导致每次请求都 Redis miss 后继续查数据库。比如大量随机不存在的商品 ID，如果没有保护，缓存层挡不住这些请求，数据库会被大量空查询消耗。它和缓存击穿不同：击穿是存在的热点 key 过期后并发回源，穿透是数据本来就不存在。

空值缓存适合挡住重复访问同一个不存在 key 的流量。数据库确认不存在后，Redis 写入一个短 TTL 的 null marker；后续请求命中 null marker 就直接返回不存在，不再查数据库。TTL 要短，避免后续数据被创建出来后，旧空值缓存仍然让用户查不到。

如果是随机大量不存在 ID，空值缓存效果会变差，甚至会把 Redis 塞满大量 null marker。此时要靠更前置的防线：参数校验挡明显非法 ID，布隆过滤器判断 ID 是否可能存在，网关和风控按 IP、用户、设备等维度限流或拒绝异常扫描流量。

Bloom Filter 判断“可能存在”后，数据库仍然可能查不到，因为 Bloom Filter 有假阳性。这个时候不是 Bloom Filter 失效，而是它的正常边界。DB 返回不存在后，应该写短 TTL 空值缓存，挡住后续同一个 ID 的重复访问。

### 缓存穿透和缓存击穿的区别

缓存穿透访问的是不存在的数据，请求没有被 Redis 挡住，一路查到数据库，数据库也返回不存在，浪费了数据库资源。缓存击穿访问的数据通常真实存在，只是热点 key 在某一刻过期或被删除，大量请求同时 miss 并发回源数据库。两者都会增加数据库压力，但根因不同：穿透要解决“无效请求进入数据库”，击穿要解决“热点 key 并发回源”。

复盘结论：已掌握缓存穿透的核心主线。缓存穿透是请求的数据在缓存和数据库里都不存在，导致请求绕过缓存持续打到数据库。治理要按链路分层：网关/API 做限流、风控和参数校验；Bloom Filter 拦截一定不存在的随机 ID；Redis 用正常缓存和短 TTL 空值缓存挡住重复不存在 key；DB 查不到后写 null marker。空值缓存适合重复不存在 key，但不适合随机大量 ID，否则会污染 Redis。Bloom Filter 有假阳性，“可能存在”仍要继续查 Redis/DB；新商品创建后要同步或异步加入 Bloom Filter，删除数据则通过定期重建、Counting Bloom 或 deleted denylist 控制假阳性积累。
