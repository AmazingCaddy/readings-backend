# 第二课：HTTP 超时与重试

原文：[HTTP 超时与重试](../../fundamentals/http-timeout-retry.md)

## 核心理解

超时负责“别无限等”，重试负责“临时失败再试一下”，幂等负责“重试不会造成重复副作用”。没有超时，慢下游会长期占用调用方线程、协程、连接和队列资源；没有边界的重试，会在下游变慢时把局部故障放大成全链路故障。

## 短回答背后的解释

### 为什么没有超时会拖住调用方资源

Product API 调用 Inventory API 时，如果没有超时，调用方会一直等待下游返回。等待期间，请求并不是“什么都不消耗”，它会占住 Product Service 的处理资源，例如线程或协程、HTTP client 连接、请求上下文、内存和队列位置。上游客户端也拿不到响应，只能继续等待。如果这种请求变多，Product Service 自己的资源会被慢下游拖满，即使其他不依赖库存服务的接口也可能被影响。

### 为什么不同阶段需要不同 timeout

一次 HTTP 调用不是一个单一动作，而是由多个阶段组成：先从连接池拿连接；如果没有空闲连接，可能要 DNS 解析、建立 TCP 连接、做 TLS 握手；请求发出去后，还要等待下游处理并返回响应。每个阶段卡住的原因不同，应该有不同边界。连接池等待超时保护调用方本地资源，connect timeout 处理下游不可达，TLS handshake timeout 处理握手卡住，read timeout 处理下游响应慢，total request timeout 保证整次调用不超过入口预算。只设置一个笼统超时，很容易让某个阶段等待过长。

### 为什么下游 timeout 不能等于入口总预算

入口 API 的 500ms 是整条请求链路的总预算，不只是库存服务的预算。这个时间里还包括网关处理、应用计算、Redis 或 MySQL、库存 RPC、响应编码、日志指标 trace，以及必要的 buffer。如果库存服务 read timeout 也设置成 500ms，一旦库存服务真的等满 500ms，入口请求基本已经没有时间完成后续处理，甚至可能入口已经超时但下游还在执行。合理做法是把入口预算拆成子预算，例如库存服务只给 100ms 到 150ms，并为其他阶段和降级留空间。

### 为什么 400 和 401 不应该重试

重试只适合“可能恢复”的临时故障，而 400 和 401 通常不是这类问题。400 表示请求参数、格式或语义有问题，同样的请求重复发送，参数不会自己变正确。401 表示未认证或认证信息无效，同样的请求重复发送，权限也不会自己恢复。对这类错误重试，只会增加调用方和下游的流量、日志噪声和排查成本。正确做法是快速失败，把错误暴露给调用方或修正请求/认证逻辑。

### 为什么 503 可以重试但不能立刻重试

503 Service Unavailable 通常表示服务暂时不可用，例如实例发布、短暂过载、依赖抖动或负载均衡摘除过程中的短暂失败。这类错误有恢复可能，所以可以有限重试。但如果所有调用方收到 503 后立刻重试，会把原始流量和重试流量同时打回正在恢复的下游，可能形成重试风暴。backoff 的作用是让重试间隔逐渐变长，给下游恢复时间；jitter 的作用是加入随机抖动，避免大量请求在同一时刻再次打过去。

### backoff 和 jitter 的区别

backoff 解决的是“不要马上重试”的问题：第一次失败后等 50ms，第二次等 100ms，第三次等 200ms，让重试间隔逐渐变长，给下游恢复时间。jitter 解决的是“不要大家同时重试”的问题：在 backoff 的基础上加入随机抖动，例如 100ms 变成 100ms 到 130ms 之间的随机值。这样即使很多请求同一时间失败，它们的下一次重试也会被打散，而不是在同一个时间点再次冲击下游。

### 为什么创建订单超时后不能直接重试

POST /orders 是有副作用的写操作，请求超时并不代表服务端一定没有执行成功。可能服务端已经创建了订单，只是响应在返回途中超时；也可能服务端还在处理中；也可能确实失败了。如果客户端或服务端直接用同一个业务动作再发一次，而后端没有幂等 key、唯一约束或请求去重机制，就可能创建两笔订单。正确做法是让客户端或调用方携带稳定的幂等 key，例如 order_request_id，后端用这个 key 保证同一个业务请求只产生一次效果；重试时如果发现已经处理过，就返回同一笔订单结果。

### 写请求应该怎样设计重试策略

读请求通常没有副作用，重试失败的 GET 更多是在重复读取数据，所以可以对连接错误、短暂超时、429、503 做有限重试，并配合 backoff、jitter 和重试预算。写请求不同，它会改变状态，例如创建订单、扣库存、扣款、发券。写请求的安全策略不是“绝不重试”，而是“先让重试安全”。常见做法是调用方生成稳定幂等 key，后端用唯一索引或幂等表记录处理状态；如果第一次已经成功，重试返回同一结果；如果正在处理中，返回处理中或让调用方查询结果；如果明确失败，才允许重新发起或进入补偿流程。对于支付、扣款这类强副作用操作，通常还要配合状态机、回调幂等和对账补偿。

### 为什么同一个 Idempotency-Key 要返回同一笔订单

Idempotency-Key 表示“这是同一次业务意图”。客户端第一次下单超时后重试，并不代表用户想买两次，而是想确认刚才那次下单有没有成功。后端发现同一个 key 已经创建过订单时，应该返回之前的订单结果，而不是再创建一笔。这样无论调用方重试几次，最终业务效果都是一笔订单。工程上通常会给幂等 key 建唯一索引，记录处理状态和关联订单 id，避免并发重试时穿透幂等保护。

### 第一次请求处理中时，第二次重试怎么处理

同一个 Idempotency-Key 的第一次请求还没处理完，第二次请求已经来了，这时不能让两个请求同时创建订单。常见做法是幂等表记录状态：PROCESSING、SUCCESS、FAILED。第二次请求看到 PROCESSING，可以直接返回 202 Accepted 或 409 Conflict，并提示稍后查询；也可以在服务端短暂等待第一次请求完成，然后返回同一结果。选择哪种取决于接口体验和系统复杂度，但核心原则是不并发执行两次写操作。处理中状态还需要过期和恢复机制，避免第一次请求崩溃后 key 永远卡住。

### 为什么多层重试会乘法放大 QPS

如果入口是 1000 QPS，每个请求对下游最多尝试 3 次，那么下游最坏会看到 3000 QPS。更危险的是多层服务都重试：A 调 B 最多 3 次，B 调 C 也最多 3 次，那么 C 最坏会看到 1000 * 3 * 3 = 9000 QPS。这说明重试不是局部行为，它会改变整条链路的流量形态。服务链路越深，每层都独立重试越危险，所以需要统一重试策略、重试预算、熔断和限流，避免下游在故障时被重试流量压垮。

### 重试预算怎么实现

重试预算通常按下游服务、接口或调用方维度统计。简单实现是滚动窗口计数：在最近 10 秒或 1 分钟内记录 original_count 和 retry_count，如果 retry_count 小于 original_count * budget_ratio，就允许重试，否则快速失败或降级。另一种实现是 token bucket：每个原始请求按比例生成少量 retry token，例如 10% 预算就是每个原始请求生成 0.1 个 token；每次重试消耗 1 个 token，token 不足就不重试。token bucket 可以设置容量上限，允许少量突发，但不会无限积累。工程上可以本地每实例近似实现，也可以用 Redis/sidecar 做全局预算；很多场景本地近似已经足够，因为目标是限制重试风暴，而不是做精确计费。

## 面试表达

如果 Product API 调用库存服务没有设置超时，而库存服务卡住，Product API 会一直等待下游返回。这个等待会占住调用方的线程/协程、HTTP 连接、请求上下文和队列资源，也无法及时返回给上游。并发一高，慢下游就可能把调用方资源拖满，所以跨服务调用必须设置明确的超时边界。

HTTP 调用通常不只需要一个总超时，而要按阶段设置边界。比如连接池等待超时、DNS/connect timeout、TLS handshake timeout、read timeout 和 total request timeout。因为这些阶段代表不同问题：可能是本地连接池满了，可能是下游不可达，也可能是下游处理慢。分开设置才能快速定位和保护资源。

入口 API 的总预算要拆给各个阶段，不能让某个下游调用吃完整个预算。比如入口总预算是 500ms，库存服务 read timeout 不应该也设置成 500ms，而应该给库存调用一个更小的子预算，同时给应用计算、缓存/数据库、响应返回、日志观测和 buffer 留时间。否则下游一旦等满，入口就会跟着超时，系统也很难优雅降级。

重试应该用于临时故障，例如连接 reset、短暂网络错误、429 或 503，而不是用于确定性的客户端错误或权限错误。503 可以做有限重试，但不能立刻无脑重试，要配合最大重试次数、backoff、jitter、重试预算和熔断。

POST /orders 这类创建订单的写操作不能因为超时就直接重试，必须先看后端是否支持幂等。安全做法是由调用方传入稳定的幂等 key，后端基于幂等 key 或唯一约束做去重，保证同一次业务请求重复提交时只创建一笔订单，并返回一致结果。

读请求和写请求最大的区别是副作用。GET /products/1001 这类读请求通常可以对临时错误做有限重试，并配合 backoff、jitter 和重试预算；POST /orders 这类写请求不能无脑重试，必须先设计幂等 key、唯一约束或幂等表，保证重复提交只产生一次业务效果。

重试会放大下游流量。入口 1000 QPS，如果每个请求最多 1 次原始调用加 2 次重试，下游最坏会看到 3000 QPS。如果调用链上多层服务都重试，放大是乘法关系，例如 A 调 B 最多 3 次、B 调 C 最多 3 次，C 最坏会看到 9000 QPS。因此重试必须有限次数，并配合重试预算、限流和熔断。

库存精确数量在商品详情里通常是弱依赖，不应该拖垮主链路。因此 Product API 会给库存 RPC 设置更短 timeout，重试次数也更保守；如果库存服务失败，就用 Redis 库存快照、模糊库存文案或隐藏精确库存做降级。技术策略要服务业务优先级，非核心信息要允许降级。
