# 第四课：数据库索引与慢查询

原文：[数据库索引与慢查询](../../database/index-and-slow-query.md)

## 核心理解

索引不是“给字段加个目录”这么简单，它要和查询条件、排序方式、分页方式、数据分布、写入频率、连接池和事务锁一起设计。慢 SQL 不只会让自己慢，还会延长连接持有时间，进而导致连接池等待、API P99 升高和重试放大。

## 短回答背后的解释

### 为什么没有合适索引会接近 O(N)

如果查询条件是 `WHERE user_id = 42`，但数据库没有能按 `user_id` 快速定位的索引，它就可能需要扫描大量订单行，逐行判断 user_id 是否匹配。表有 5000 万行时，这种扫描接近 O(N)。如果还要 `ORDER BY created_at DESC LIMIT 20`，数据库可能还要从匹配结果里排序，才能拿到最近 20 条。最终返回只有 20 行，但中间可能读了大量无用数据，导致 SQL 变慢、连接持有时间变长，并把连接池和接口 P99 一起拖慢。

### 为什么单列 user_id 索引不一定够

单列 `user_id` 索引能帮助数据库更快找到某个用户的订单，但它只解决了过滤问题。如果这个用户有很多订单，数据库仍然可能需要取出大量 `user_id = 42` 的记录，再按 `created_at DESC` 排序，最后取前 20 条。也就是说，过滤路径有了，但排序路径不一定匹配。更好的索引通常要同时服务过滤和排序，例如 `(user_id, created_at DESC)`，让数据库能在这个用户的订单范围内直接按创建时间倒序扫描，拿到前 20 条就停下来。

### 为什么等值过滤字段通常放在复合索引前面

对于 `WHERE user_id = 42 AND status = 'PAID' ORDER BY created_at DESC LIMIT 20`，更自然的复合索引是 `(user_id, status, created_at DESC)`。`user_id` 和 `status` 都是等值条件，放在前面可以先把扫描范围缩小到“这个用户的已支付订单”。在这个小范围内，索引再按 `created_at DESC` 排列，数据库就可以顺序读出最近 20 条。如果把 `created_at` 放在最前面，数据库会优先按全表时间顺序扫描，再检查 user_id 和 status，可能要跳过大量不属于这个用户或状态不匹配的记录。

### 为什么分页排序需要稳定 tie-breaker

`created_at` 不是唯一字段，同一秒甚至同一毫秒可能有多条订单。如果只按 `created_at DESC` 排序，这些相同时间的记录之间没有确定顺序。第一页和第二页查询时，数据库可能用不同顺序返回同一批相同时间的记录，导致分页重复或遗漏。解决方式是在排序里加一个唯一且稳定的字段作为 tie-breaker，例如 `ORDER BY created_at DESC, id DESC`，对应索引也加上 `id DESC`。这样即使时间相同，也能用 id 决定唯一顺序。

### 为什么索引不是越多越好

索引本质上是数据库额外维护的数据结构，是用空间和写入成本换读取速度。查询时，合适索引可以减少扫描和排序；但写入、更新和删除时，数据库不仅要改数据行，还要同步维护相关索引。索引越多，写入放大越明显，占用的磁盘和内存也越多。对高频写入表，如果随便给每个字段都建索引，可能读查询没优化多少，写入吞吐先下降了。因此索引要围绕核心查询模式设计，而不是看到字段就建。

### 为什么低选择性字段不适合单独索引

`is_deleted` 只有 true/false 两种值，选择性很低。单独建索引时，如果大多数数据都是 `is_deleted = false`，数据库通过这个索引仍然会命中大量行，过滤效果很差，甚至可能不如直接扫描。软删除字段通常会出现在几乎所有业务查询里，但它更适合作为复合索引的一部分，例如 `(user_id, is_deleted, created_at DESC)` 或 `(tenant_id, is_deleted, status, created_at DESC)`，配合更高选择性的字段一起缩小范围。

### 如何理解 EXPLAIN 里的 ALL、rows 和 filesort

在 MySQL 里，`type = ALL` 通常表示全表扫描，是比较差的访问方式；`rows = 50000000` 表示优化器预计要扫描 5000 万行；`Using filesort` 表示排序没有直接利用索引顺序，而需要额外排序。三者放在一起，说明这条 SQL 的过滤条件和排序条件大概率没有匹配到合适索引。排查时要回到查询模式：WHERE 条件是什么，ORDER BY 是什么，LIMIT/OFFSET 怎么用，再设计或调整复合索引，并用 EXPLAIN 验证 `type`、`key`、`rows` 和 `Extra` 是否改善。

### 为什么走了索引也可能慢

走索引只说明数据库选择了某个索引访问路径，不代表扫描的数据量一定小。比如低选择性字段、范围过大的条件、复合索引字段顺序不匹配，都可能导致数据库虽然走了索引，但仍然扫描几十万甚至几百万条索引记录。除此之外，如果查询字段不在索引里，还可能发生大量回表；如果排序没有被索引覆盖，还可能继续 filesort。所以看 EXPLAIN 不能只看 `key` 是否有值，还要看 `rows`、过滤比例、Extra、回表成本和实际执行耗时。

### 为什么 OFFSET 深分页会慢

`LIMIT 20 OFFSET 100000` 的语义不是直接跳到第 100001 条，而是先按查询条件和排序找到前 100020 条，再丢弃前 100000 条，返回后面的 20 条。页数越深，数据库处理的无用数据越多。即使走了索引，也可能要扫描大量索引记录；如果还需要回表或排序，成本会更高。深分页慢查询会让连接持有时间变长，进一步拖高连接池等待和接口 P99。

### cursor pagination 为什么更适合深翻页

cursor pagination 的核心是记住上一页最后一条记录的位置，例如 `(created_at, id)`。下一页查询不再说“跳过前 100000 条”，而是说“从上一页最后一条之后继续查”。例如倒序分页时使用 `created_at < last_created_at OR (created_at = last_created_at AND id < last_id)`，再 `ORDER BY created_at DESC, id DESC LIMIT 20`。这样数据库可以沿着复合索引从游标位置继续向后扫描，直接返回下一页需要的记录，避免大量无效跳过。

### cursor pagination 的代价是什么

Cursor pagination 更适合“下一页/上一页”这种连续浏览，不适合直接跳到第 100 页。因为第 100 页的位置取决于前面每一页的最后一条游标，除非系统额外保存页码到游标的映射、做搜索快照或维护专门的分页索引，否则不能像 OFFSET 那样直接表达“跳过前 1980 条”。另外，多次翻页之间数据集可能变化：新订单插入到更前面，后续页面通常不会看到它；旧订单被删除，后续页面数量可能变化。Cursor 通常比 OFFSET 更稳定，但如果业务要求严格一致的分页快照，就需要额外的快照版本、固定查询时间点或导出任务。

### 什么是覆盖索引

覆盖索引是指查询需要的字段都包含在索引里，数据库可以直接从索引返回结果，不需要再回到数据表读取完整行。比如索引是 `(user_id, status, created_at DESC, id DESC)`，查询只返回 `id, user_id, status, created_at`，这些字段都在索引里，就可能形成覆盖索引。如果查询还要返回 `amount, address, remark`，但这些字段不在索引里，数据库就需要先通过索引找到主键，再回到数据表读取完整行，这个动作叫回表。覆盖索引能减少回表和随机 IO，但会让索引变大，增加写入维护成本。

### 为什么覆盖索引不能无限塞字段

覆盖索引的目标是减少回表，但不是把整张表复制进索引。如果把 `amount, address, remark, coupon_info` 等大量字段都塞进索引，索引会变得很大，占用更多磁盘和内存，缓存命中率也可能下降。写入、更新、删除时，数据库还要维护这个大索引，写入成本会上升。尤其是长文本、JSON、地址、备注这类大字段，更不适合为了覆盖而放进索引。覆盖索引更适合高频、核心、返回字段少且字段较小的查询。

### 订单列表慢查询怎么形成闭环排查

订单列表 P99 升高并伴随连接池等待升高时，要把现象串起来看：慢 SQL 持有连接更久，连接归还变慢，连接池 pending 增多，API P99 被拖高。排查时先确认连接池等待是否真的升高，再聚合慢 SQL 模板，找到高频慢查询。对 `WHERE user_id = ? ORDER BY created_at DESC LIMIT 20 OFFSET ?` 运行 EXPLAIN，看是否全表扫描、扫描行数过多、Using filesort 或深分页扫描。修复不只是加索引，还要处理深分页：可以增加 `(user_id, created_at DESC, id DESC)` 这类复合索引，并把 OFFSET 改为 cursor pagination。上线后继续观察 SQL P95/P99、扫描行数、连接池 acquire time 和 API P99。

### 订单列表状态筛选的复合索引

对于 `WHERE user_id = ? AND status = ? ORDER BY created_at DESC, id DESC LIMIT 20`，复合索引可以设计为 `(user_id, status, created_at DESC, id DESC)`。`user_id` 和 `status` 是等值过滤，放在前面缩小范围；`created_at DESC, id DESC` 和排序一致，可以让数据库在过滤后的范围内按索引顺序读取结果。`id DESC` 作为 tie-breaker，保证同一时间的订单排序稳定，也能配合 cursor pagination 使用 `(created_at, id)` 作为游标。

## 面试表达

如果一个 5000 万行的订单表要查某个用户最近 20 条订单，但没有匹配 `user_id` 和 `created_at` 的索引，数据库可能需要扫描大量行，再过滤和排序。虽然最终只返回 20 条，但中间处理的数据量很大，延迟会随着表规模增长而恶化。慢 SQL 还会占住数据库连接，导致连接池等待和接口 P99 升高。

只给 `user_id` 建单列索引不一定能完全优化 `WHERE user_id = ? ORDER BY created_at DESC LIMIT 20`。它能帮助过滤某个用户的订单，但如果这个用户订单很多，数据库仍然可能需要对这些订单按 `created_at` 排序。更合适的是设计复合索引，比如 `(user_id, created_at DESC)`，让过滤和排序使用同一条索引路径。

对于多个等值过滤加排序的查询，复合索引通常把等值过滤字段放前面，把排序字段放后面。例如 `WHERE user_id = ? AND status = ? ORDER BY created_at DESC LIMIT 20` 可以考虑 `(user_id, status, created_at DESC)`。这样数据库先定位到某个用户某个状态的订单范围，再按索引里的创建时间顺序直接取前 20 条，避免额外排序和大量无效扫描。

分页查询的排序必须稳定。如果只按 `created_at DESC` 排序，而很多订单的 created_at 相同，数据库对这些记录的返回顺序可能不稳定，翻页时就可能出现重复或遗漏。通常会加上主键作为 tie-breaker，例如 `ORDER BY created_at DESC, id DESC`，索引也设计成 `(user_id, status, created_at DESC, id DESC)`。

索引不是越多越好。它是典型的空间换时间：合适索引可以加速查询，但每次 insert、update、delete 都要维护相关索引，索引越多写入成本越高，也会占用更多存储和内存。实际设计时应该从核心访问模式出发，只给高价值查询建立合适的单列索引、复合索引或覆盖索引。

`is_deleted` 这类 true/false 字段通常不适合单独建索引，因为选择性太低，单独过滤时仍会命中大量数据。它更适合放进符合业务查询路径的复合索引里，配合 user_id、tenant_id、status、created_at 等字段一起使用。

如果 EXPLAIN 显示 `type = ALL`、`rows = 50000000`、`Extra = Using filesort`，我会认为这是慢查询高风险信号。它说明数据库可能在全表扫描 5000 万行，并且排序没有利用索引顺序，需要额外排序。下一步会检查 WHERE、ORDER BY 和分页方式是否有匹配的复合索引，并用 EXPLAIN 验证扫描行数和排序方式是否下降。

EXPLAIN 显示走了索引不代表 SQL 一定没问题。还要看这个索引扫描了多少行、过滤后剩多少行、是否发生大量回表、是否还有 filesort 或 temporary。一个查询即使用了索引，如果索引选择性很差或范围太大，仍然可能扫描大量索引记录，最终表现为慢查询。

OFFSET 深分页会慢，是因为数据库通常不是直接跳到第 N 条，而是先找到并跳过前面的 N 条记录。例如 `LIMIT 20 OFFSET 100000` 可能要先处理 100020 条，再丢掉前 100000 条，只返回 20 条。页数越深，无效扫描越多，所以高并发接口更适合 cursor pagination。

cursor pagination 的核心是用上一页最后一条记录的排序键作为游标，让下一页从这个位置之后继续查。例如按 `created_at DESC, id DESC` 排序时，下一页携带 `(last_created_at, last_id)`，SQL 通过游标条件过滤掉前面的数据，再取 `LIMIT 20`。这样避免了 OFFSET 深分页从头跳过大量记录的问题。

Cursor pagination 的代价是它不擅长任意跳页，更适合连续加载下一页。它依赖上一页最后一条记录的游标，所以不能天然直接跳到第 100 页。并且多次请求之间如果数据发生新增或删除，看到的结果不一定是同一个时间点的快照；如果业务要求严格一致，需要引入快照时间、版本号或额外的分页索引设计。

覆盖索引是指查询需要的字段都包含在索引里，数据库可以直接从索引返回结果，不需要再回表读取完整数据行。它能减少回表和随机 IO，提高查询性能。但覆盖索引会让索引变大，也会增加写入和更新时的维护成本，所以只适合高频、核心、返回字段较少的查询。

覆盖索引不能为了避免回表就无限塞字段。索引过大会占用更多存储和内存，降低缓存效率，也会增加 insert、update、delete 的维护成本。设计覆盖索引时要克制，优先服务高频核心查询，并尽量只包含必要的小字段。

订单列表接口 P99 升高，同时连接池等待也升高时，我会先确认是不是慢 SQL 持有连接太久导致连接归还变慢。接着聚合慢 SQL 模板，用 EXPLAIN 看扫描行数、是否全表扫描、是否 filesort，以及是否存在深分页 OFFSET。修复方向通常是增加匹配查询条件和排序的复合索引，例如 `(user_id, created_at DESC, id DESC)`，并把 `OFFSET` 深分页改成 cursor pagination。上线后要观察 SQL P95/P99、扫描行数、连接池等待和接口 P99 是否下降。

如果订单列表按 `user_id` 和 `status` 过滤，并按 `created_at DESC, id DESC` 排序，我会设计 `(user_id, status, created_at DESC, id DESC)` 这样的复合索引。前两个字段用于等值过滤，后两个字段用于稳定排序和游标分页，避免额外排序和深分页扫描。

## 复盘记录

| 复盘题 | 回答 | 结果 |
| --- | --- | --- |
| 为什么索引不是越多越好？ | 索引越多维护成本越高，占用的存储也越高，本质是空间换时间，只在高频的字段做索引优化 | 通过。索引要围绕高价值访问模式设计，不能看到字段就建。 |
| 为什么 `is_deleted: true/false` 这种字段通常不适合单独建索引？什么时候可以放进联合索引？ | 这是低选择性字段，过滤出来的行数还是会有很多 | 通过。单独索引收益低；当它和 user_id、tenant_id、status、created_at 等核心查询条件一起使用时，可以放进复合索引。 |
| 为什么 `LIMIT 20 OFFSET 100000` 会慢？它是不是直接跳到第 100001 条？ | 不是直接跳到 100001 条，而是查出了 100020 条，再一个个数过去的 | 通过。OFFSET 深分页会处理并丢弃大量无用记录，页数越深越慢。 |
| cursor pagination 相比 OFFSET pagination，核心思想是什么？有什么代价？ | 利用之前查询的位置，过滤掉大量行数，减少处理的行数。代价是不能直接去某一页 | 通过。Cursor 更适合连续翻页，不擅长任意跳页，也不天然保证多次请求看到同一个快照。 |
| `WHERE user_id = ? AND status = ? ORDER BY created_at DESC, id DESC LIMIT 20` 应该设计什么复合索引？为什么顺序合适？ | `index(user_id, status, created_at desc, id desc)`。先把 where 过滤条件加进来，减少行数，然后利用索引排序，取出来的前 20 个就是最终答案 | 通过。等值过滤字段在前，排序和游标字段在后，支持范围缩小、稳定排序和 LIMIT 早停。 |

复盘结论：已掌握索引与慢查询的基础主线。索引要围绕访问模式设计，不是越多越好；复合索引字段顺序要匹配等值过滤、排序和分页；EXPLAIN 不能只看是否走索引，还要看扫描行数、filesort、temporary 和回表；深分页 OFFSET 会处理大量无用记录，高并发列表接口更适合 cursor pagination。
