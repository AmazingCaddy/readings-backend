# 第八课：Cache-Aside 模式

原文：[Cache-Aside 模式](../../cache/cache-aside.md)

## 核心理解

Cache-Aside 是最常见的缓存模式：应用先读缓存，缓存 miss 后读数据库，再把结果写回缓存。缓存是加速层，数据库才是权威数据源。它适合读多写少、允许短暂不一致、miss 后能从数据库恢复的数据；但缓存一致性、缓存穿透、击穿、雪崩和热点 key 都需要额外设计。

## 短回答背后的解释

### 为什么商品详情适合 Cache-Aside

商品详情通常是读多写少的数据：用户会频繁查看商品名称、图片、价格摘要、描述和配置，但商品基础信息不会每次请求都变化。如果每次请求都查数据库，高 QPS 会消耗数据库连接池、CPU、IO 和 buffer pool，慢查询或连接等待还会推高接口 P99。使用 Cache-Aside 后，应用先查 Redis；命中时直接返回缓存数据，不访问数据库；miss 时再查数据库并写回缓存。这样能把大部分高频读请求挡在缓存层，让数据库主要承担首次加载、缓存过期后的回源和写入事务。

### 为什么不存在的数据也可以缓存

如果某个 `product_id` 在数据库里不存在，而每次请求都因为 Redis miss 去查数据库，攻击流量、爬虫流量或客户端错误重试都可能把大量“不存在查询”打到数据库。这类问题叫缓存穿透的基础场景。常见做法是缓存一个空值标记，例如 `__null__`，表示数据库确认不存在。下次再查到这个 key 时，应用直接返回 not found，不再访问数据库。空值缓存的 TTL 要短，例如几十秒，因为这个商品以后可能被创建出来；如果空值缓存太久，用户会继续看到不存在。

### TTL 太短和太长的取舍

TTL 决定缓存能存在多久。TTL 太短时，缓存很快过期，同一个商品会频繁 miss 并回源数据库，缓存命中率下降，数据库压力和接口延迟都会上升。TTL 太长时，缓存命中率会提高，但数据新鲜度变差；如果商品详情、价格、上下架状态已经更新，用户可能长时间读到旧缓存。实际配置要结合业务变化频率和一致性要求：变化少、允许短暂旧值的数据可以长一些；价格、库存展示、权限等更敏感的数据要更短，或者配合写后删缓存、版本号、主动失效等机制。

### 为什么写路径通常是更新数据库后删除缓存

Cache-Aside 的缓存由应用在读路径上懒加载，所以写路径更常见的做法是先更新数据库，提交成功后删除对应缓存。删除缓存的好处是简单，写路径不需要重新组装完整缓存对象，也不需要处理部分字段更新、序列化结构变化、多个缓存 key 同步更新等复杂问题。下一次读取时 Redis miss，应用会从数据库读到最新值并重新写回缓存。代价是下一次读会少一次缓存命中，但换来更简单、更不容易出错的一致性处理。

### Cache-Aside 为什么会有短暂不一致

Cache-Aside 不是强一致协议。写路径更新数据库后，如果删除 Redis 失败，旧缓存还在，后续读请求会继续命中旧值，直到 TTL 自然过期或后续删除成功。并发读写也可能产生旧值回填：读请求先 miss 并查到旧数据库值，写请求随后更新数据库并删除缓存，读请求最后又把旧值写回缓存。对于允许短暂旧值的商品详情，可以依赖较短 TTL 做最终兜底；如果业务对新鲜度更敏感，就要增加删除重试、延迟双删、版本号校验、消息/binlog/CDC 驱动失效，或者在关键读路径绕过缓存读数据库。

### 为什么 TTL 要加 jitter

如果大量商品缓存都使用固定 TTL，例如统一 5 分钟过期，那么在某个时间点同一批 key 可能同时失效。失效后大量请求同时 Redis miss，流量会集中回源数据库，造成数据库连接池、CPU、IO 压力陡增，接口 P99 也会升高。TTL jitter 是在基础 TTL 上增加随机抖动，例如 `300s + random(0, 60s)`，让不同 key 的过期时间分散开。它不能解决单个热点 key 过期瞬间的击穿问题，但能降低大量 key 同时过期带来的雪崩风险。

### Redis 故障时为什么不能全量回源

Cache-Aside 的数据库是权威数据源，但这不代表 Redis 故障时所有流量都可以直接打到数据库。缓存平时承担了大量高频读流量，Redis 一旦不可用，如果应用把所有请求都回源数据库，数据库连接池、CPU 和 IO 可能瞬间被打满，导致本来只是缓存故障，最后演变成数据库和核心写入一起故障。更合理的策略是按数据库承受能力做回源限流，只允许一部分请求查数据库；非核心请求可以快速失败或降级；热点数据可以用本地缓存、静态快照或默认值兜底；同时对 Redis 访问做短超时和熔断，避免请求长时间卡在缓存层。

### Cache-Aside 不适合什么数据

Cache-Aside 适合读多写少、允许短暂旧值、miss 后能从数据库恢复的数据，例如商品详情、配置项、用户展示资料。它不适合作为强一致数据的唯一读取依据，例如余额、账务、账户权限、订单支付状态这类数据。如果这些数据读到旧值，可能造成错误扣款、越权访问或错误履约。强一致数据即使用缓存，也通常要非常谨慎：关键判断读数据库或事务内校验，缓存只用于展示或非关键路径；或者引入版本号、主动失效、CDC、短 TTL 等机制。但如果为了保证一致性把 TTL 设置得极短、频繁失效并不断回源，缓存收益会很低，还增加工程复杂度和 Redis 压力，不如直接读权威数据源。

### 缓存命中率低怎么排查

商品详情用了 Cache-Aside 但 Redis 命中率只有 40%，说明大量请求仍然在回源数据库。第一步可以看 key 设计是否合理：同一个商品是否因为语言、渠道、用户态、无序参数、版本字段等拼出了多个不同 key，导致命中率被打散。第二步看 TTL 是否太短，缓存频繁过期；第三步看 miss 后是否真的成功写回 Redis，是否存在序列化失败、写缓存超时或异常被吞掉；第四步看是否有大量不存在的商品 id，需要空值缓存；第五步看 Redis 内存和淘汰策略，是否 key 刚写入就被 evict；最后结合 Redis 错误率、GET/SET 延迟、keyspace hit ratio、DB QPS 和接口 P99 判断瓶颈。

### 为什么不要把用户态字段混进商品详情缓存

商品基础详情是商品维度的数据，例如名称、图片、类目、描述；“用户是否收藏过该商品”是用户和商品之间的关系，属于 user-product 维度。如果把收藏状态直接放进 `product:{id}`，同一个商品对不同用户的结果不同，要么缓存内容会错误复用给其他用户，要么 key 必须变成 `product:{id}:user:{userId}`。后者会把一个高复用的商品缓存拆成大量用户级 key，命中率明显下降，缓存占用和失效复杂度也会上升。更合理的做法是商品基础详情单独缓存，用户收藏状态单独查询或单独缓存，例如 `favorite:{userId}:{productId}`，最后在接口层组合结果。

### Cache-Aside 的读路径

Cache-Aside 的读路径由应用控制。请求进来后，应用先根据业务 id 构造缓存 key，例如 `product:1001`，然后读 Redis。如果命中正常数据，就反序列化并直接返回，不访问数据库。如果 Redis miss，就回源查询数据库；数据库查到记录后，把结果写回 Redis，并设置 TTL 和 jitter，再返回给调用方。如果数据库也查不到，可以写入短 TTL 的空值标记，避免不存在数据反复穿透到数据库。读路径的关键是：缓存只是加速层，miss 后一定能从数据库恢复，数据库才是权威来源。

## 面试表达

商品详情这类接口通常适合加 Redis 缓存，因为它读多写少，且很多字段可以接受短暂不实时。如果每次请求都打数据库，高 QPS 会消耗连接池、CPU 和 IO，甚至影响其他核心写入。使用 Cache-Aside 后，应用先读缓存，命中就直接返回；缓存 miss 才查数据库并写回缓存，从而降低数据库压力和接口延迟。

数据库查不到的数据也可以做空值缓存。否则不存在的 product_id 每次都会缓存 miss 并回源数据库，恶意请求或错误流量会持续消耗数据库资源。通常会缓存一个 null marker，并设置较短 TTL，既能挡住重复不存在查询，又避免后续真实数据创建后长时间被旧空值挡住。

TTL 的设置是在命中率和数据新鲜度之间取舍。TTL 太短会导致缓存频繁失效，读请求不断回源数据库，达不到保护数据库和降低延迟的目的；TTL 太长会让旧数据存在更久，用户可能在商品已经更新后继续看到旧详情。实际设计要按业务容忍度选择 TTL，并配合写后删除缓存来缩短不一致窗口。

Cache-Aside 写路径通常不是“更新数据库后直接更新缓存”，而是“更新数据库后删除缓存”。删除缓存更简单可靠，下一次读取会 miss 并从数据库重新加载最新数据。直接更新缓存看起来能保留命中率，但在复杂对象、多个缓存 key、并发写入和部分字段更新时更容易引入不一致。

Cache-Aside 只能做到最终一致或短暂不一致，不是强一致。更新数据库成功后，如果删除缓存失败，旧缓存还会被读到；并发读写交错时，也可能出现旧值被重新写回缓存。TTL 可以作为兜底让旧缓存最终失效；如果业务对一致性要求更高，需要删除重试、延迟双删、版本号或 CDC 失效等机制。

缓存 TTL 通常要加 jitter，避免大量 key 在同一时间过期。如果所有商品缓存都固定 5 分钟过期，某一批缓存同时失效时，大量请求会同时 miss 并回源数据库，导致数据库压力陡增。加随机抖动可以把过期时间打散，降低缓存雪崩风险。

Redis 故障时不能让所有商品详情请求无保护地回源数据库。否则缓存层承载的高频读流量会瞬间转移到数据库，导致连接池等待、SQL 延迟和 P99 全部升高，甚至影响核心写入。工程上要按数据库容量做回源限流，只放行一部分请求，其他请求降级、快速失败或使用本地缓存/快照兜底。

Cache-Aside 不适合所有数据。它适合商品详情这类读多写少、允许短暂旧值的数据；余额、权限、支付状态这类强一致数据不能简单依赖缓存。关键业务判断通常应该读数据库或做事务内校验。如果为了保证一致性把缓存 TTL 设置得极短，缓存命中率低、复杂度高，还不如直接访问权威数据源。

如果 Cache-Aside 上线后命中率仍然很低，我会先查 key 设计是否把同一个业务对象拆成了多个缓存 key，例如参数顺序、渠道、语言或用户态导致 key 不一致。然后看 TTL 是否太短、miss 后是否成功写回、是否有大量不存在 key、Redis 是否超时或发生淘汰，以及缓存容量是否足够。目标是先解释为什么没有命中，再决定是调整 key、TTL、空值缓存、容量还是回源保护。

商品基础详情和用户态字段不应该混在同一个缓存对象里。比如“是否收藏”是 user-product 维度，如果放进 `product:{id}`，要么会把某个用户的状态错误返回给其他用户，要么只能把 key 拆成用户级 key，导致命中率下降和缓存膨胀。更合理的是商品详情按 product 维度缓存，收藏状态按用户关系单独查询或缓存，最后在接口层组合。

Cache-Aside 的读路径是：先根据业务 id 构造 key 读 Redis，命中就直接返回；miss 后查数据库，查到就把结果写回缓存并设置 TTL 和 jitter；查不到则写短 TTL 的空值缓存，避免不存在数据反复打数据库。整个过程中数据库是权威数据源，缓存只是加速层。

| 为什么“不存在的数据”也要做空值缓存？空值缓存 TTL 为什么不能太长？ | 虽然不存在，那也是查了一次数据库之后才知道不存在，这样还是会浪费数据库资源；如果有恶意攻击、爬虫软件等都会拖垮数据库，让 Redis 先挡一下流量。TTL 不能太长，否则真有这个 key 的数据被创建出来之后，会查不到 | 通过。空值缓存能挡住不存在 key 的重复回源，防止缓存穿透；短 TTL 能减少后续数据创建后仍被旧空值挡住的时间。 |
| Redis 故障时，为什么不能让所有请求都直接回源数据库？怎么保护数据库？ | Redis 的吞吐量一般比数据库高很多，Redis 故障之后，如果请求都直接走数据库，数据库肯定受不了；可以分一部分流量到数据库，其他流量降级处理 | 通过。Redis 故障不能全量回源，要按数据库容量做限流，只放行部分请求，其余快速失败、降级或使用本地缓存/快照。 |
| Cache-Aside 适合什么数据？不适合什么数据？ | 适合读多写少的数据，内容比较稳定，不怎么更新，一致性要求不高。商品详情这种更新较少的数据适合，余额和权限这种敏感数据，以及一致性要求高的数据不适合 | 通过。Cache-Aside 适合商品详情、配置、展示资料等允许短暂旧值的数据；余额、权限、支付状态等强一致数据不能简单依赖缓存。 |

复盘结论：已掌握 Cache-Aside 的核心主线。读路径是先查 Redis，命中直接返回，miss 后查数据库并写回缓存；数据库查不到时可以写短 TTL 空值缓存。写路径通常是先更新数据库，再删除缓存，让下一次读重新加载。Cache-Aside 能降低读多写少场景的数据库压力，但只能做到最终一致或短暂不一致；TTL 要在命中率和新鲜度之间取舍，并加 jitter 防止大量 key 同时过期。Redis 故障时不能全量回源数据库，要做限流、降级或本地缓存兜底。它适合商品详情这类允许短暂旧值的数据，不适合余额、权限、支付状态这类强一致数据的关键判断路径。
