# 第三课：连接池

原文：[连接池](../../fundamentals/connection-pool.md)

## 核心理解

连接池是一组可复用的网络连接，用来减少频繁建连成本，并限制应用同时打到数据库、Redis 或下游服务的并发。连接池不是越大越好：太小会导致请求等待连接，太大会把下游连接数、并发查询和资源调度压力打满。

## 短回答背后的解释

### 为什么每次请求都新建连接很贵

数据库连接不是普通函数调用，它背后有网络连接、认证、协议握手，有些场景还包括 TLS。每次请求都新建连接，会让调用方和数据库反复做建连和释放工作。低 QPS 时可能只是多一点延迟，高 QPS 时会产生大量短连接，数据库要花资源处理连接建立、认证、内存分配和关闭，而不是专心执行查询。连接池通过复用已有连接，把这部分固定成本摊薄，同时也限制了同时进入数据库的并发数量。

### 为什么连接池不是越大越好

连接池不仅是复用连接的工具，也是控制下游并发的闸门。数据库能高效并行处理的查询数量是有限的，超过这个范围后，并发更多不一定更快，反而会增加上下文切换、锁竞争、内存占用、磁盘 IO 排队和查询等待。如果数据库稳定能承受 100 个并发查询，应用把连接池开到 200，并且瞬时真的放进 200 个查询，数据库可能整体变慢，所有请求 P99 都升高。连接池调大只能减少应用侧等待，不会凭空增加数据库处理能力。

### 为什么池等待高不能立刻大幅调池

等待数据库连接 1200ms、SQL 查询 80ms，说明当前慢点发生在借连接阶段，而不是这条 SQL 的执行阶段。但这不自动等于“把池调大就能解决”。要先看连接池 active connections 是否长期等于 max、pending requests 是否升高、connection acquire time 是否变长；再看数据库 CPU、锁等待、慢 SQL、事务耗时和最大连接数。如果连接被长事务、慢查询或泄漏占住，单纯调大池会让更多请求进入数据库，可能把数据库拖慢。合理做法是先定位连接为何不释放，再在数据库容量允许时小步调大，并继续观察 P95/P99、DB CPU 和 pool wait。

### 为什么看到 connection timeout 要先澄清含义

日志里的 connection timeout 可能指不同问题：等待连接池里的空闲连接超时，说明本地池可能耗尽；建立数据库或 HTTP 连接超时，说明下游不可达、网络慢或连接建立失败；也可能是客户端库把多种 timeout 统一包装成 connection timeout。排查时要先看错误来源、异常类型和 trace span，确认是 pool wait timeout、connect timeout 还是 read/query timeout。只有确认是池等待超时后，才按连接池耗尽方向继续看 active、idle、pending、acquire time、慢 SQL、长事务、连接泄漏、流量突增和数据库容量。

### active 打满但 DB CPU 不高说明什么

active connections 长期等于 max、pending requests 高，说明应用侧连接池被占满，新请求在等连接。但数据库 CPU 不高，说明瓶颈不一定是数据库计算能力耗尽。可能是连接被代码借走后没有归还，也可能是长事务或慢 SQL 持有连接很久但并不消耗太多 CPU，例如锁等待、IO 等待、深分页扫描、外部依赖等待期间仍占着事务连接。也可能只是流量突增或池配置过小。排查时要看连接持有时间、事务耗时、慢 SQL、lock wait、连接泄漏检测日志和应用代码的 release/finally 路径。

### 为什么异常路径必须归还连接

连接池的前提是“借了会还”。如果正常路径归还连接，但异常路径忘记 release，连接池会出现连接泄漏。泄漏不是一次性报错，而是可用连接逐渐减少：一开始只是偶发等待，随后 active connections 长期接近 max、idle 变少、pending requests 增加，最后大量请求等待连接超时。因为数据库 CPU 可能并不高，这类问题容易被误判成连接池太小。所有语言都要用 try/finally、try-with-resources、defer Close 或 context manager 保证连接一定归还。

### 为什么慢 SQL 会打满连接池

连接池能承载多少请求，不只取决于连接数量，还取决于每个请求持有连接多久。如果 SQL 从 50ms 变成 800ms，单个连接的周转速度会明显下降。请求进入速度不变，但连接归还速度变慢，active connections 会逐渐升高并达到 max，新请求只能排队等待连接。于是一个慢 SQL 不仅让自己的查询变慢，还会让其他请求在连接池前等待，最终表现为 API P99 升高和 connection acquire time 增加。

### 为什么池等待超时要小于入口预算

入口 API 的 500ms 是整次请求的总预算，不只是等待数据库连接的预算。如果连接池等待超时设置成 1s，请求可能在池前排队 1 秒才失败或拿到连接，但此时上游用户请求早就超过 500ms 预算了。即使最终拿到连接，也已经没有时间执行 SQL、编码响应和记录观测数据。池等待超时应该是入口预算中的一个子预算，例如几十到一两百毫秒，池满时快速失败、降级或限流，而不是排队到入口超时。

### 如何按实例数估算连接池大小

连接池大小不能只看单个实例，要看所有实例加起来会给数据库多少连接。如果数据库最多希望承受 200 个应用连接，Product API 有 10 个实例，平均每个实例就是 20 个连接。实际配置通常还要给其他服务、后台任务、运维连接、连接波动和扩容留余量，所以每实例可能控制在 15 到 20 左右。不能每个实例都配置 30，因为 10 个实例就是 300 个连接，已经超过数据库目标容量。这个估算只是起点，最终还要结合压测、DB CPU、IO、锁等待、连接等待和查询耗时调整。

### 为什么深分页会拖垮连接池

`LIMIT 20 OFFSET 100000` 这类深分页查询通常需要数据库扫描或跳过大量行，再返回后面的少量结果。即使最终只返回 20 行，数据库也可能做大量排序、扫描和 IO。查询变慢后，连接被持有的时间变长，连接归还速度下降，active connections 更容易打满，新请求开始 pending。此时连接池等待高是结果，不是根因。正确方向是优化分页方式，例如使用合适索引、基于游标/created_at + id 翻页、限制最大页数，必要时做搜索/归档方案，而不是直接把连接池调大让更多慢查询同时进入数据库。

## 面试表达

### 连接池为什么不是越大越好

连接池大小要看数据库本身的承受能力，连接池不仅是复用连接的工具，也是对数据库的一种保护。池太小会让应用侧请求等待连接，导致 P99 升高；但池太大也不是好事，因为它会允许更多查询同时进入数据库，超过数据库能高效处理的范围后，会带来上下文切换、锁竞争、内存占用、IO 排队和查询等待。连接池大小应该从数据库容量、服务实例数和接口并发倒推，并通过压测和监控逐步调整。

当前自己的简短回答：要看数据库本身的承受能力，连接池的大小，是对数据库的一种保护。

如果 trace 显示等待数据库连接 1200ms、SQL 查询 80ms，我会判断瓶颈主要在连接池等待，而不是这条 SQL 本身。但我不会直接把连接池从 16 调到 100，而是先看 active、idle、pending、connection acquire time、连接泄漏、长事务、慢查询和数据库容量。如果确认数据库还有余量且只是池配置偏小，可以小步调大连接池；如果是连接被慢事务或泄漏占住，就要先修复根因。

看到大量 connection timeout 时，我不会立刻判断连接池太小，而是先确认 timeout 发生在哪个阶段：是等待池连接超时、建连超时，还是查询/read 超时。确认是池等待超时后，再看 active connections、pending requests、connection acquire time、连接泄漏、长事务、慢 SQL、流量突增和数据库容量。只有确认数据库还有余量且池配置偏小，才会小步调大连接池。

如果连接池 active 长期等于 max、pending 很高，但数据库 CPU 不高，我会怀疑连接被长时间占用而不是数据库算力打满。常见原因包括连接泄漏、长事务、慢 SQL 或锁等待、流量突增、池配置偏小，以及应用在持有连接期间做了不必要的外部等待。下一步会看连接持有时间、事务耗时、慢 SQL、lock wait、泄漏检测日志和代码是否可靠 release。

借到连接后必须用 finally release、defer Close 或类似机制保证归还。否则异常路径一旦忘记释放，就会发生连接泄漏，池里的可用连接越来越少，最终 active 长期打满、pending 请求增多，大量请求在等待连接时超时。连接泄漏通常要通过连接持有时间、泄漏检测日志和代码审查来定位。

慢 SQL 会导致连接池被打满，因为连接被持有的时间变长，归还速度下降。即使请求 QPS 不变，原来每个连接 50ms 就能归还，现在要 800ms 才归还，连接周转率大幅降低，active connections 更容易达到 max，新请求就会在池前排队，API P99 也会升高。

如果连接池 active 和 pending 都很高，同时慢 SQL 里出现大量深分页查询，我会优先怀疑 SQL/分页方式导致连接持有时间变长，而不是直接判断连接池太小。深分页会让数据库扫描或跳过大量行，查询变慢后连接归还变慢，进而造成池等待。修复方向是优化索引、改成游标分页、限制深分页或引入更适合查询的检索方案。
