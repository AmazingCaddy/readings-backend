# 第一课：请求生命周期

原文：[一个请求的完整生命周期](../../fundamentals/request-lifecycle.md)

## 核心理解

前端/客户端通常看到的是“发请求、等响应、渲染页面”。后端需要把这次请求拆成一条链路：网关、应用服务、Redis、MySQL、下游服务、日志、指标和 trace。接口变慢时，不能只猜业务代码或 SQL 慢，而要定位请求到底在哪一段排队、等待、超时或重试。

## 短回答背后的解释

### 为什么“网络卡顿”是合理第一反应，但不能停在这里

从前端/客户端视角看，页面偶尔慢，最容易先想到网络卡顿，这是合理的，因为用户侧确实会受到弱网、DNS、TCP/TLS 建连、丢包、响应下载和客户端本地处理影响。但后端排查不能只停在“网络慢”，因为一次接口请求是一条链路：客户端、网络、网关、应用服务、Redis、MySQL、下游服务、日志和 trace。任何一段排队、等待或重试，都会让用户感觉慢。所以第一步不是直接下结论，而是先拆链路。

### 为什么客户端 2s、后端 80ms 时优先怀疑网络或入口链路

客户端看到 2s，说明用户完整等待了 2 秒；后端日志显示 80ms，说明后端应用真正处理请求的时间很短。两者之间差出来的 1920ms，大概率不在业务代码、Redis、MySQL 或下游 RPC 里，而是在请求进入后端前、后端返回后，或者网关/传输层之间。此时应该优先查看客户端 timing、DNS/TCP/TLS、网关日志、负载均衡日志、响应大小和客户端本地处理，而不是一上来优化 SQL。

### 为什么客户端 2s、后端 1800ms 时要继续拆后端内部

如果客户端总耗时 2s，后端处理已经占了 1800ms，说明主要慢点在后端内部。但“后端接口慢”还不是根因，只是定位范围。下一步要用 trace 或分段日志继续拆：应用计算是否慢，线程池或协程是否排队，Redis 是否 miss 或连接池等待，MySQL 是否慢查询、锁等待或连接池耗尽，下游 RPC 是否超时，是否发生了重试放大。后端排查的关键是继续问“这 1800ms 花在哪里”。

### 为什么等待数据库连接 1200ms、SQL 80ms 时先看连接池

这类问题很容易被误判成“数据库慢”，但 trace 已经说明真正 SQL 查询只用了 80ms，主要耗时在等待数据库连接。也就是说，请求还没真正开始执行 SQL，就已经在连接池前排队了。常见原因包括连接池太小、并发突然升高、连接泄漏、事务持有连接太久、慢请求占满连接池，或者上游重试把请求量放大。此时优先排查连接池指标、活跃连接数、等待队列、等待超时、事务耗时和连接释放逻辑。

### 为什么库存服务 RPC 1500ms 时优先看库存链路

trace 的价值是把后端 1800ms 拆成多个阶段。如果应用计算、Redis、MySQL 和响应返回都很短，而库存服务 RPC 占了 1500ms，那么优先怀疑库存链路是合理的。但这还不等于“库存服务代码一定有问题”。RPC 慢可能发生在多个位置：调用方等待 HTTP 连接、网络传输、库存服务网关排队、库存服务内部查库、库存服务调用它自己的下游，或者调用方超时后又重试了一次。下一步要看库存服务自己的 trace、错误率、P95/P99、连接池等待、超时和重试配置。

### 为什么重试可能让系统更危险

重试不是天然有益，它是在用更多请求换取一次成功机会。对于偶发网络抖动、连接重置、短暂 429/503 这类临时错误，受控重试可能提高成功率。但如果下游已经变慢、连接池已经满、线程已经排队，重试会把一次请求变成两次甚至多次请求，直接增加下游 QPS、连接占用和排队时间。结果可能是原本只有库存服务慢，后来商品服务线程也被占住，网关队列变长，更多请求超时，形成级联故障。所以重试必须有最大次数、短超时、退避、jitter、熔断和幂等保障，并且不是所有错误都应该重试。

### 为什么平均延迟正常时仍然要看 P99

平均延迟描述的是整体平均水平，但它会把少数极慢请求摊薄。比如 99 个请求都是 50ms，1 个请求是 2s，平均值可能仍然看起来不算糟，但那个 2s 的用户体验是真实存在的。P99 表示最慢 1% 请求的延迟，更适合发现长尾问题。长尾通常来自偶发资源等待，例如连接池排队、锁等待、缓存 miss 后回源、下游 RPC 超时、GC 暂停、重试或热点 key。用户说“偶尔很慢”时，平均延迟往往不够敏感，应该优先看 P95/P99、慢请求 trace 和各依赖分段耗时。

### 为什么看 trace 要先看每一步的时间

trace 的核心价值不是告诉我们“这个接口慢”，而是把一次请求拆成多个 span，并标出每个阶段的耗时。第一眼看每一步的时间，是为了找到主要耗时段：慢在应用计算、Redis、MySQL、下游 RPC、连接池等待、锁等待，还是重试。找到最长阶段后，还要继续下钻。例如 MySQL 慢要区分 SQL 执行慢、锁等待慢、连接池等待慢；RPC 慢要区分调用方等待连接、网络慢、下游入口排队、下游内部依赖慢。不要只看总耗时，要看耗时分布。

## 面试表达

### 一个接口偶尔很慢，怎么排查

我的第一反应不是直接猜 SQL 慢，而是先确认这是长尾问题还是整体变慢。会先看 P95/P99，取出 P99 慢请求日志，找到对应的 trace id。然后对比客户端总耗时、网关日志和后端 access log，先判断慢是在网络/入口链路/客户端侧，还是后端接口内部。如果客户端很慢但后端处理很快，就优先查网络、DNS/TCP/TLS、网关传输和客户端侧；如果后端处理本身也慢，就看 trace 里面每个 span 的时间分布，优先找耗时最长的阶段，再继续拆应用计算、Redis、MySQL、连接池、锁等待、下游 RPC、超时和重试，一层层定位根因。

当前自己的简短回答：先找 P99，取出 P99 的 log，找到慢的 trace id，先排查是否是网络问题还是接口问题；如果是接口问题，排查详细 trace 里面的时间分布，优先看耗时最长的，一步步拆解下去。
