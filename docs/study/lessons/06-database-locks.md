# 第六课：数据库锁

原文：[数据库锁](../../database/database-locks.md)

## 核心理解

锁是保证并发写入正确性的工具，但锁范围越大、持有越久，性能风险越高。排查慢接口时，要区分 SQL 执行慢、锁等待慢和连接池等待慢。SQL 本身可能很快，但一直在等别人释放锁。

## 短回答背后的解释

### 为什么更新同一行会等待

当事务 A 执行 `UPDATE orders SET status = 'PAID' WHERE id = 1001` 时，数据库会对这行加排他锁，防止其他事务同时修改同一行造成状态互相覆盖。事务 A commit 或 rollback 前，这把锁不会释放。事务 B 如果也想更新 `id = 1001`，通常就要等待事务 A 结束。如果等待太久，可能触发 lock wait timeout。锁等待期间，事务 B 可能占着数据库连接和应用请求上下文，进而影响连接池和接口 P99。

### 为什么事务里调用外部 HTTP 会放大锁等待

如果事务 A 更新订单后，在事务内调用外部 HTTP 并等待 2 秒才 commit，那么订单行的排他锁也会多持有 2 秒。事务 B 如果想更新同一行，就会一直等事务 A 提交或回滚。这个等待不是 B 的 SQL 复杂，而是被 A 的长事务阻塞。外部 HTTP、MQ、慢计算这类不可控等待不应该放在数据库事务里，否则会让锁持有时间变长，连接归还变慢，接口 P99 升高，甚至引发 lock wait timeout。

### 如何区分 SQL 慢、锁等待和连接池等待

如果 trace 显示 SQL 执行只有 20ms，connection pool wait 只有 5ms，但 lock wait 是 1500ms，那么瓶颈在锁等待。也就是说，请求很快拿到了数据库连接，SQL 真正执行也很快，但大部分时间花在等其他事务释放锁。排查时应优先找锁持有者：哪个事务持有锁、持续了多久、执行了什么 SQL、是否在事务里调用外部 HTTP/MQ、是否有长事务或未提交事务。不能把这种问题误判成 SQL 需要加索引或连接池太小。

### 死锁是怎么发生的

死锁发生在事务之间形成环形等待。比如事务 1 先锁账户 A，再想锁账户 B；事务 2 先锁账户 B，再想锁账户 A。此时事务 1 等事务 2 释放 B，事务 2 等事务 1 释放 A，两个事务都无法继续。数据库通常会检测到死锁，并选择其中一个事务作为 victim 回滚，释放它持有的锁，让另一个事务继续。应用需要捕获死锁错误，并按业务安全性有限重试整个事务。

### 为什么固定锁顺序能降低死锁概率

如果所有转账事务都按统一规则加锁，例如先锁账户 id 小的，再锁账户 id 大的，那么 A 转 B 和 B 转 A 实际都会先竞争同一把锁。一个事务拿到第一把锁后，另一个事务会在第一把锁上等待；拿到锁的事务继续锁第二个账户、完成更新、提交释放锁。这样等待关系是单向排队，不容易形成“事务 1 持有 A 等 B、事务 2 持有 B 等 A”的环形等待。固定锁顺序不能消灭所有锁等待，但能显著降低多资源并发更新时的死锁概率。

### 为什么 pool wait 高不一定先调连接池

如果 trace 同时显示 lock wait 很高、SQL execution 很短、connection pool wait 也升高，通常要先判断连接池等待是不是被锁等待间接放大的。锁等待会让数据库连接被请求占用更久，连接归还速度下降；在请求量不变时，后续请求拿连接就会排队，pool wait 自然升高。此时直接调大连接池可能只是让更多请求同时进入数据库，反而扩大锁竞争和数据库压力。更合理的优先级是先查长事务、热点行、事务内外部调用、范围锁和死锁，再决定是否需要调整连接池。

### 为什么索引会影响锁范围

更新语句不只是查数据，还会对要修改的数据加锁。如果 `UPDATE orders SET status = 'EXPIRED' WHERE created_at < ? AND status = 'CREATED'` 没有合适索引，数据库可能需要扫描大量行来判断哪些记录符合条件。扫描和更新范围变大后，被加排他锁的行会更多；在 MySQL InnoDB 的一些范围查询场景下，还可能出现间隙锁或 next-key lock，阻塞同范围内的插入或更新。合适的索引能让数据库更快定位小范围数据，从而减少扫描量和锁影响范围。

### 死锁后为什么可以有限重试

数据库检测到死锁后，通常会选择一个事务回滚，释放它已经持有的锁，让另一个事务继续。被回滚的事务如果直接失败，用户可能看到偶发错误；但很多死锁是瞬时并发冲突，等另一个事务提交后，重试整个事务往往可以成功。这里要注意两点：第一，重试的是整个事务逻辑，而不是只重试最后一条 SQL，因为事务前面的读取和判断也可能已经失效；第二，重试必须有限次数，并且要有幂等、唯一约束或状态机保护，避免重复下单、重复扣款、重复发消息这类副作用。

### 如何排查批量更新导致的锁等待

如果下单接口 P99 突然升高，trace 显示 lock wait 远高于 SQL execution 和 connection pool wait，同时最近上线了“批量把超时未支付订单改成 EXPIRED”的任务，应优先怀疑批量更新扩大了锁影响范围。排查顺序可以是：先确认慢请求是否都卡在订单表锁等待；再看批量 UPDATE 的 WHERE 条件是否有合适索引，例如 `status`、`expire_at` 或 `created_at` 的组合索引；再看任务是否一次更新太多行、事务是否过长、是否和下单/支付回调争抢同一批订单行。常见优化是用索引缩小扫描范围，按主键或时间窗口分批更新，每批尽快提交，并控制后台任务并发和执行时间窗口。

## 面试表达

两个事务同时更新同一行时，先执行更新的事务通常会持有这行的排他锁。后一个事务如果也要更新同一行，需要等待前一个事务 commit 或 rollback 后才能继续。这个等待不是 SQL 本身执行慢，而是锁等待；如果事务很长，就会导致后续请求排队、连接被占住，最终推高接口 P99。

如果事务里更新数据后又调用外部 HTTP，事务提交会被外部调用拖慢，锁也会被持有更久。其他事务如果要更新同一行，就会一直等待，甚至触发锁等待超时。所以事务应该尽量短，只包含必要的数据库读写，外部调用应放到事务外或通过可靠异步机制处理。

如果 trace 里 SQL 执行时间很短、连接池等待也很短，但 lock wait 很长，我会判断瓶颈是锁等待。下一步不是优先优化 SQL 或调连接池，而是查谁持有锁、事务持续时间、是否有长事务、事务内是否做了外部调用或慢计算，以及相关 SQL 是否锁住了过大的范围。

死锁通常来自多个事务以不同顺序加锁，形成互相等待。比如 A 转 B 的事务先锁 A 再锁 B，B 转 A 的事务先锁 B 再锁 A，就可能各自持有一把锁并等待对方释放。数据库会检测死锁并回滚其中一个事务，应用需要处理这个错误，必要时有限重试整个事务。

为了降低死锁概率，多个事务需要锁同一批资源时，应该约定固定加锁顺序。比如所有转账都按账户 id 从小到大锁，这样并发事务会在同一把第一锁上排队，而不是各自拿住一把锁后互相等待。这个设计把死锁风险转成普通锁等待，系统行为更可控。

如果 trace 里 connection pool wait 和 lock wait 都很高，但 SQL execution 很短，我不会第一反应就调大连接池。我会先判断是不是锁等待导致事务变长、连接占用时间变长，进而让连接池归还变慢。优先排查谁持有锁、是否有长事务、热点行、事务内外部调用或范围锁。锁等待降低后，连接占用时间缩短，pool wait 往往也会跟着下降。

没有合适索引的范围更新，风险不只是 SQL 慢，还可能扩大锁范围。数据库为了找到符合条件的行，会扫描更多记录；更新命中的行需要加排他锁，范围条件在某些隔离级别下还可能带来间隙锁。线上排查这类问题时，我会结合 EXPLAIN、锁等待指标和当前事务，判断是不是扫描范围太大导致锁住了过多数据。

数据库检测到死锁后，通常会回滚其中一个事务。应用层需要识别这类可重试异常，对整个事务做有限重试，因为另一个事务完成后，锁冲突可能已经消失。但重试前要确认业务是幂等的，或者有唯一索引、状态机、幂等 key 等保护，避免重试带来重复扣款、重复下单或重复发送消息。

如果下单接口 P99 突然升高，而 trace 显示 lock wait 最高，我会先判断瓶颈是锁等待。结合最近上线的批量过期订单功能，我会重点检查这条批量 UPDATE 是否缺少合适索引、是否一次更新太多行、事务是否过长，以及是否和下单或支付回调争抢订单行。处理方式通常不是先调大连接池，而是缩小锁范围和锁持有时间，例如增加合理组合索引、分批更新、每批快速提交，并限制后台任务并发。

排查数据库锁问题时，我会重点看锁等待时间、长事务数量和事务持续时间，以及当前到底是谁持有锁、谁在等待锁。如果锁等待很高，还要进一步看是否有事务内外部调用、批量更新、缺索引范围更新、热点行更新或固定锁顺序缺失。锁等待变长后会让连接被占用更久，所以还要结合连接池等待一起看，避免误判成单纯连接池太小。
